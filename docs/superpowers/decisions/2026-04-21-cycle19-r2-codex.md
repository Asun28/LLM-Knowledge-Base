### M1 â€” Manifest key must cover both manifest writes (reservation + confirmation)
File/line: `src/kb/ingest/pipeline.py:1062-1093` (duplicate reservation), `src/kb/ingest/pipeline.py:1301-1310` (tail confirmation)
Issue: `ingest_source` computes `source_ref = make_source_ref(...)` once, then uses it for both the phase-1 duplicate reservation and the phase-2 confirmation. Adding `manifest_key=` only at the final write leaves the reservation under the old key, so a compile-driven ingest can temporarily create two manifest entries if it fails after reservation â€” or duplicate-detection observes the old key and misses an already-ingested source.
Suggested fix: Derive `manifest_ref = manifest_key if manifest_key is not None else source_ref` once at the top of `ingest_source` and pass it into `_check_and_reserve_manifest` and the tail confirmation. Keep `source_ref` unchanged for page frontmatter and logs unless the design explicitly wants those to reflect the compile key.

### M2 â€” Pending refine-history rows have no recovery policy
File/line: `src/kb/review/refiner.py:36-41` (history load/save), `docs/superpowers/decisions/2026-04-21-cycle19-brainstorm.md:114-117` (two-phase write discussion)
Issue: The proposed two-phase history write makes `pending` visible in the JSON store. If `atomic_json_write` succeeds for the pending write and the process dies before flipping to `applied`, the row stays `pending` forever with no operational signal beyond raw JSON inspection. The brainstorm does not commit to a sweep or recovery contract.
Suggested fix: Cycle 19 does not need a full sweep tool. At minimum: (a) document that `pending` is terminal-forensic until a later explicit sweep, (b) add a small testable helper (e.g., `list_stale_pending(hours=24)`) that reports pending entries older than a threshold so operators can detect stuck rows. Defer mutation/cleanup to a follow-on cycle if scope is tight.

### M3 â€” Batch injection must re-derive the winning target under the per-page lock
File/line: `src/kb/compile/linker.py:216-242` (pre-lock match selection), `src/kb/compile/linker.py:252-296` (under-lock write), `src/kb/ingest/pipeline.py:1333-1351` (inject_wikilinks call site)
Issue: The existing single-target path peeks page content without a lock, then re-reads and re-validates under the per-page lock before writing. A batch implementation that selects the winning target during the pre-lock scan (from a preloaded `all_wiki_pages` snapshot) can make a stale decision: a concurrent ingest may have already inserted the link or updated the page content between the snapshot and the lock acquisition. The lock still serializes writes, but the chosen target or link text may be wrong.
Suggested fix: Treat the pre-lock scan only as a candidate-gathering pass. Under the page lock, re-read the file, recompute existing links and candidates for the full current chunk, then choose the deterministic winner from fresh content and write at most one link per title.

### M4 â€” Windows tilde-path test leaves the platform-neutral canonicalization gap untested
File/line: `docs/superpowers/decisions/2026-04-21-cycle19-requirements.md:78-80` (AC13 tilde case), `src/kb/compile/compiler.py:50-63` (`detect_source_drift` / manifest key construction)
Issue: The tilde-short-path case is Windows-specific and the stated default is to skip it off Windows. That misses the platform-neutral half of the same bug: divergent canonicalization when `raw_dir` is non-default, relative, or symlinked differently between `compile_wiki` and `ingest_source`.
Suggested fix: Keep the Windows tilde test behind `pytest.mark.skipif(not sys.platform.startswith("win"), ...)`. Add a portable sibling test that uses a custom `raw_dir` under a temp project with a relative or symlinked spelling (e.g., `pathlib.Path(tmp_path / "link").symlink_to(raw_dir)`) and asserts both call sites resolve to the same manifest key.

### M5 â€” ReDoS cap is count-bounded but not per-title length-bounded
File/line: `docs/superpowers/decisions/2026-04-21-cycle19-requirements.md:56` (MAX_INJECT_TITLES_PER_BATCH=200), `src/kb/compile/linker.py:202-207` (regex construction), `src/kb/ingest/pipeline.py:910-916` (batch item validation)
Issue: A 200-title chunk is safe for typical 2-60 character titles (alternation payload ~â‰¤12 KB). But `_process_item_batch` validates item type/emptiness only â€” it does not cap individual title length before `re.escape`. One 100 KB extracted entity title can still produce a very large alternation branch and slow every page scan in the chunk. `re.escape` does not deflate length; worst case the regex segment for that title is 2Ă— its character count.
Suggested fix: Add a per-title length guard before building the alternation (e.g., skip or warn on titles >500 chars). Keep the 200-count chunk cap because it bounds normal cases. Add a dedicated test for a single pathologically long title to confirm it is rejected or truncated cleanly.

### N1 â€” `manifest_key` placement must stay after the existing `*` sentinel
File/line: `src/kb/ingest/pipeline.py:964-973` (function signature)
Issue: The current signature already has `*` before `defer_small`, `wiki_dir`, `raw_dir`, and `_skip_vector_rebuild`. Placing `manifest_key` before `*` would create a new positional slot â€” a silent break for MCP callers that already pass `source_path, source_type` positionally.
Suggested fix: Use `*, defer_small=False, wiki_dir=None, raw_dir=None, manifest_key=None, _skip_vector_rebuild=False` and add a regression test that verifies legacy positional calls bind identically after the change.

### N2 â€” MCP module-attribute lookup is not a meaningful hot-path perf concern
File/line: `src/kb/mcp/core.py:48-59` (import style), `src/kb/mcp/core.py:312-379` (ingest tool), `src/kb/mcp/core.py:489-526` (query tool)
Issue: Switching `from kb.ingest.pipeline import ingest_source` to `pipeline.ingest_source(...)` adds one attribute lookup per call. Those calls immediately perform LLM/API work, disk ingest, or BM25 search â€” the lookup is not measurable relative to the actual work.
Suggested fix: Adopt the owner-module style for patchability (as designed). The real risk is incomplete migration leaving some call sites on the old import. Add one vacuity test per callable family (or a parametrized test) that proves a monkeypatch on the owner module intercepts `query_wiki`, `search_pages`, `compute_trust_scores`, and `ingest_source` at the MCP tool layer.

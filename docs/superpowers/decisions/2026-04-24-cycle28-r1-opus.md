## Analysis

### AC1 — `_ensure_conn` perf-counter + INFO log

**Testability.** Strong. `caplog` fixture at `INFO, logger="kb.query.embeddings"`
captures the record; a monkeypatched `sqlite_vec.load` stub satisfies the fast
path. The assertion shape is the same as cycle-26 test 4 which captures the
`"Vector model cold-loaded in"` INFO record. Revert-divergence: drop the
`logger.info(...)` line and the assertion flips. Drop the `start =
time.perf_counter()` line and the `elapsed` evaluation errors out — divergent
at runtime, not just at assertion time. No signature-only drift risk.

**Semantic correctness.** The AC prose says "BEFORE returning the connection",
which matches `_ensure_conn`'s current structure at line 491: `self._conn =
conn; return conn`. Inserting a `start = perf_counter()` before line 471
(`sqlite3.connect`) and an `elapsed = perf_counter() - start` + `logger.info`
between line 491 and 492 keeps the contract. One wrinkle: the AC prose
brackets "the sqlite_vec extension load" specifically, but the natural bracket
also includes `sqlite3.connect` and `enable_load_extension` which are cheap
(<10ms typical). Requirements should clarify: is the reported elapsed
specifically the `sqlite_vec.load(conn)` call, or the full block from
`sqlite3.connect` through `enable_load_extension(False)`? The brainstorm says
"brackets the extension-load block" — ambiguous. I recommend bracketing the
FULL block (line 471-476) for two reasons: (1) `sqlite_vec.load` is the
dominant cost so the attribution is ~correct anyway, (2) narrower brackets
create a refactor hazard if a future cycle re-orders the `connect`/`load`
calls. Flag this as a CONDITION to lock in at Step 5.

**Risk map.** Maps to T1 (log-injection via `db_path`) and T2 (path
disclosure). Both accepted as existing-pattern parity per cycle-26 precedent
(`embeddings.py:138` already logs `vec_path` at INFO unredacted). No cycle-11
L2 revert-tolerant risk because the test uses a branching assertion ("INFO
record matching this format exists"), not an inspect.getsource grep.

**Assumptions.** (a) `time` is importable at module level or function-local —
cycle 26 uses function-local `import time` (line 360) and that precedent works;
cycle 28 should match. See Concern 6. (b) `self.db_path` never contains an ANSI
escape — true under the filesystem trust model. (c) `caplog` captures INFO
records without manual handler attachment — standard pytest behavior; confirmed
by cycle-26 test 4 pattern.

### AC2 — `SQLITE_VEC_LOAD_WARN_THRESHOLD_SECS = 0.3` constant + threshold WARNING

**Testability.** Strong. Same `monkeypatch.setattr(time, "perf_counter",
fake)` pattern as cycle-26 test 4 where a 0.5s stub produced both records.
Cycle 28 test wants two branches: one above threshold (WARNING fires) and one
below (WARNING absent) — both are deterministic with monkeypatched `perf_counter`.
Revert-divergence: remove the `if elapsed >= THRESHOLD:` guard and one of the
two tests fails.

**Semantic correctness.** The value 0.3 matches cycle-26 `VECTOR_COLD_LOAD_WARN_THRESHOLD_SECS`.
Question: is 0.3s defensible for sqlite-vec extension load? See Concern 5 —
my rough-order estimate is that a cold sqlite-vec DLL load on Windows NTFS
without filesystem cache is in the 50-200ms range; on warm machines typically
<50ms. 0.3s is therefore a reasonable warn threshold (won't fire on healthy
machines, will fire if filesystem is hot or DLL-cache cold). Worth calling out
that this threshold is shared-precedent-by-coincidence with cycle-26's model
cold-load (which can easily exceed 0.3s during first HF-Hub fetch). A future
cycle may split them if operator feedback shows one or the other is mis-tuned.

**Risk map.** T5 (WARN DoS) explicitly addressed — `_ensure_conn` is locked
to ≤1 call per `VectorIndex` instance, cache capped at `MAX_INDEX_CACHE_SIZE =
8`, so warn rate bounded. Matches cycle-16 L2 test-behavior-not-signature: the
test exercises a FAKE `perf_counter` that produces both branches, not a
source-grep.

**Assumptions.** Threshold semantics match cycle-26 (`>=` inclusive comparison
at line 369). Zero-elapsed case (monkeypatched to return constant 0) — does
not fire WARNING because `0.0 >= 0.3` is false. Safe.

## Specific-concern findings

**1. Is `_conn_lock` held across the WHOLE extension-load block?** YES.
Verified by Read of `embeddings.py:453-492`: `with self._conn_lock:` at line
453 wraps everything from the double-check at 455 through the terminal
`return conn` at 492. A `+= 1` and `logger.info(...)` inserted between lines
491 and 492 sit inside the lock. Requirements AC3 prose is correct.

**2. Cycle-25 `_dim_mismatches_seen` lock-free pattern causing test-ordering
bugs across test modules?** LOW RISK. Verified by Read of
`test_cycle26_cold_load_observability.py:176` which uses the monotonic-delta
pattern (`baseline = get_X(); ... assert delta == 1`) — reload-safe. Cycle-20
L1 reload-leak class applies to "absolute count" assertions, NOT delta
assertions. Cycle 28 AC6 tests MUST clone this pattern. Grep verification at
Step 9: `grep "get_sqlite_vec_load_count\b" tests/test_cycle28_*.py` should
show every call-site paired with a `baseline` or `before` snapshot.

**3. Existing test spies on `BM25Index` that would break on unconditional
`logger.info`?** NO. Grep confirms `test_bm25_nonempty_corpus_no_warning` at
`test_lint_query_fixes_v092.py:233` uses `caplog.at_level(logging.WARNING,
logger="kb.query.bm25")` — it captures only WARNING+ records. AC4 emits INFO,
not WARNING, and the new record does NOT contain "avgdl" substring. No
collision. The 20+ other `BM25Index(...)` call-sites in the test suite do NOT
use `caplog` at all. Safe.

**4. Will `%s` for `self.db_path` accept a `pathlib.Path` cleanly?** YES.
Python's `logging` framework lazy-formats via `str()` on the underlying
object, and `Path.__str__` returns the posix/windows string repr
unambiguously. Cycle-26 at `embeddings.py:368` uses `%.2fs` only, so no
direct precedent — but `embeddings.py:138` and `:309` both log `vec_path`
(a `Path`) with `%s` successfully. The requirements at AC1 use `"db=%s" %
self.db_path` — safe. If the implementer is paranoid they can use `str(self.db_path)`
explicitly; functionally identical. Not a blocker.

**5. Is 0.3s `SQLITE_VEC_LOAD_WARN_THRESHOLD_SECS` defensible?** YES, with
one caveat. `sqlite_vec.load(conn)` calls `enable_load_extension(1)` +
dlopen of the `sqlite_vec` shared library. On a warm machine with FS cache
hot, the DLL load is ~5-30ms (SSD). On a cold machine or first boot, it can
reach 50-200ms on Windows NTFS due to Defender scan. A 0.3s threshold fires
only on pathological cold boots or heavily contested filesystem — which IS
the signal operators want. If anything the value could be tightened to 0.2s
once real telemetry lands. Keep 0.3s for cycle-28 symmetry with cycle-26's
cold-load threshold; treat tightening as a cycle 29+ tuning question.

**6. Boot-lean tests asserting `time` is NOT in `sys.modules`?** NO RISK.
Grep of `test_cycle23_mcp_boot_lean.py:66-72` shows the allowlist probes
`['anthropic', 'networkx', 'sentence_transformers', 'kb.query.engine',
'kb.ingest.pipeline', 'kb.feedback.reliability', 'kb.query.embeddings']`
— `time` is NOT in the probe list. Also, `time` is a CPython stdlib module
that is imported by countless transitive deps (including `logging` itself)
— it's already in `sys.modules` from process start. Adding `import time` at
either module scope or function-local scope in `embeddings.py` / `bm25.py`
cannot break any existing test. HOWEVER: consistency preference — cycle-26
uses FUNCTION-LOCAL `import time` at `embeddings.py:360`. Cycle 28 should
match by using function-local imports in `_ensure_conn` and
`BM25Index.__init__` bodies. This is aesthetic, not correctness.

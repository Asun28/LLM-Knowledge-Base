## Analysis

Cycle 10 is a batched hardening cycle with no net-new feature surface: it closes the cycle-9 `_validate_wiki_dir` migration on four residual MCP tools (AC7/AC8/AC9/AC10), standardises silent-degradation error surfacing via a new `_safe_call` helper in `mcp/app.py` (AC1/AC2/AC3), hardens `wiki_log.py` against torn lines on Windows (AC12/AC13), replaces a static `--- INPUT ---` fence with a cryptographically unique UUID boundary in `capture.py` (AC14), moves `captured_at` to submission time (AC15), validates extraction-field types in `ingest/pipeline.py` (AC16/AC17), adds a `kb_search` length cap + stale marker + `kb_read_page` ambiguity detection (AC4/AC5/AC6), and documents the `raw/captures/` architectural exception in CLAUDE.md (AC18). The blast radius is narrow but touches five of the eight trust-boundary files I was asked to read. Notably every ACxx below is already supported by precedent code in the module: `_validate_wiki_dir` already exists in `mcp/app.py:187-200`, `_safe_call` already ships in `kb.lint._safe_call` and is already imported by `mcp/health.py:8`, `_escape_prompt_fences` already hardens against embedded legacy fences in `capture.py:331-342`, `_sanitize_error_str` already redacts absolute paths in `mcp/app.py:145-184`, and `MAX_QUESTION_LEN` is already used by `kb_query` (`mcp/core.py:120`) and `kb_search` (`mcp/browse.py:43`). So cycle 10 is mostly *generalising* controls that already exist on neighbouring call sites.

The threat surface is therefore dominated by two concerns rather than new attack vectors. First, consistency drift: when the same four tools (`kb_stats`, `kb_graph_viz`, `kb_verdict_trends`, `kb_detect_drift`) migrate from hand-rolled `Path(wiki_dir).resolve().relative_to(PROJECT_ROOT.resolve())` to the shared helper, their pre-existing `ValueError` catch branches (see `mcp/browse.py:333-334`, `mcp/health.py:202-203`) must be removed or kept coherent so the `Invalid wiki_dir` error message does not regress into a `ValueError` path that leaks paths. The current `_validate_wiki_dir` already performs the equivalent check and returns a prefixed `Error:` string, so the replacement is net-positive if the caller threads the `(path, err)` tuple correctly. Second, new silent-degradation surfaces: `_safe_call` will widen what gets surfaced to the MCP client — every exception's `str(e)` becomes part of an `<label>_error: <str(e)>` string embedded in the tool response. This is the exact class of surface the cycle-7 redaction work (`_sanitize_error_str`) was added to defend. Cycle 10's non-AC implementation detail #2 explicitly calls out the need to reuse `_redact_paths` / `_sanitize_error_str`, but the existing `kb.lint._safe_call` (already used by `mcp/health.py:72-79` for the feedback-flagged-pages site) returns `str(e)` raw, so cycle 10 either must modify `kb.lint._safe_call` itself or wrap at the caller. The AC1 wording says the helper lives in `mcp/app.py`, not `lint/`, which implies a NEW helper — but then `mcp/health.py` will have two `_safe_call` functions unless the import is re-pointed. The PR must pick one; otherwise we get two parallel silent-degradation code paths with divergent sanitisation. Additional consideration: the two `kb.feedback.store.load_feedback` monkeypatches in AC26/AC27 will trigger OSError messages that may embed the feedback.json path; the raw `str(e)` of `OSError("corrupt json")` doesn't, but a real corruption would produce e.g. `"Expecting value: line 1 column 1 (char 0) in /home/user/…/feedback.json"` — exactly what the cycle-7 redaction exists to catch.

## Analysis

The cycle-11 requirements are well-framed backlog-by-file hardening with 14 ACs across 7 source/test files plus docs. Approach C in the brainstorm (atomic cluster → independent per-file → new tests → docs) is the right shape: it correctly calls out the 7-file import-swap cluster as a justified multi-file commit per the cycle-4 RedFlag, avoids the Approach-B test-before-refactor awkwardness (where AC6 would need to target the old location), and maps cleanly to the feedback_batch_by_file memory. The threat model is present and covers the import-path monkeypatch drift risk (T3), the silent dead-path closure (T2), the subprocess-env hygiene (T5), and the inspect-source-tests anti-pattern for AC8/AC13 (T6/T9). Cycle termination hygiene looks clean — per-file commit plan is explicit, the atomic cluster has explicit rationale, and AC14 sequences docs last. No new dependencies → CVE surface (T10) is empty by construction.

That said, I found three concrete gaps that should block or constrain the plan. **First, AC5 undercounts the same-class surface by one.** The spec says "All 5 internal callers (`compile/linker`, `evolve/analyzer`, `lint/checks`, `lint/runner`, `lint/semantic`) switch" — but `src/kb/compile/compiler.py:242-243` has its own function-local `from kb.graph.builder import page_id as get_page_id` + `from kb.graph.builder import scan_wiki_pages` inside `detect_source_drift`. If we leave this caller importing from the re-export shim, we have a 6th import-site caller silently relying on the back-compat alias — exactly the kind of scoping miss the cycle-7/8 `_validate_notes@kb_save_lint_verdict` RedFlag calls out. Second, **AC4's "delete `_page_id`" is an assumption that `grep` disproves.** Three existing test call sites import `_page_id` directly: `tests/test_v0914_phase395.py:732` (`from kb.utils.pages import _page_id as utils_page_id`) and `tests/test_v0915_task01.py:344,352` (`TestPageIdLowercase` fixture). Deleting the symbol breaks these tests immediately. The requirements' "delete the private helper" instruction must either (a) update those 3 test sites to use the new public `page_id` in the SAME atomic commit, or (b) keep `_page_id` as a deprecated alias. Not addressing this surfaces as a test-file-churn miss in Step 9, triggering the cycle-8 signature-drift verification RedFlag. Third, **AC8's adversarial-stub fixture needs a concrete plan.** The requirements' OQ5 in brainstorm says "prepend a PYTHONPATH entry containing a `kb/config.py` stub that raises at import time" — but the actual implementation needs to (i) create the stub directory layout (`<stub>/kb/config.py`) AND (ii) the stub must only shadow `kb.config`, not the whole `kb` package (which has `__init__.py`, `__version__`, etc.). Without an explicit recipe, Step 9 may implement the weaker `'kb.config' not in sys.modules` assertion that, per T6, IS behavioral — but the requirements leave this ambiguous enough that a Step 9 implementer could regress to signature-only. Threat model T6 correctly flags this, but I'd like the requirements to pick (a) or (b) before Step 9 starts so Round 2/3 doesn't debate it.

## Open Questions
- Q1: For AC4 condition 2, does the requirements author prefer "migrate 3 test sites in the same atomic commit" or "keep `_page_id` as a back-compat alias"? The brainstorm decision OQ2 picked `page_id` re-export for `kb.graph.builder` but did not discuss `_page_id` back-compat in `kb.utils.pages` itself. Given 3 test references, recommend the alias path for lowest blast radius.
- Q2: `src/kb/compile/compiler.py:242-243` uses function-LOCAL imports (not top-of-module). Should the migrated imports remain function-local (preserving the existing module-load pattern in `detect_source_drift`) or move to top-of-module? The 5 modules listed in AC5 are all top-of-module imports, so preserving the function-local pattern for compiler.py is a minor readability inconsistency but has a principled reason (avoid import cycle risk at module load).
- Q3: AC1 non-goal — the brainstorm OQ4 explicitly scopes OUT list-consumer sites (`entities_mentioned`, `concepts_mentioned` already tolerate `or []`). Is the requirements author OK with a test that injects `entities_mentioned="not-a-list"` NOT raising (silent coerce via `or []` produces empty list)? If yes, document this as a non-goal in AC1 itself, not only in brainstorm OQ4.
- Q4: AC13 Option B requires "load the manifest file before and after `compile_wiki` and assert per-key hashes match". Current test stubs `ingest_source` entirely (lines 244-255) — so no hashes are written. Does the AC13 implementation need to let the real ingest run (slow, LLM-dependent) or stub with realistic hash writes? Threat-model T9 says "parsed dict (key count + per-key hashes) is byte-equivalent" — the realistic-stub approach is fine, but requirements should say so.

## Scope/Ambiguity Flags
- F1: AC5 — "5 internal callers" is under-enumerated. Missing: `src/kb/compile/compiler.py:242-243` (function-local import inside `detect_source_drift`). Proposed clarification: update AC5 to "All 6 internal callers (`compile/compiler` function-local, `compile/linker`, `evolve/analyzer`, `lint/checks`, `lint/runner`, `lint/semantic`)".
- F2: AC4 — "The private `_page_id` helper in the same module is deleted" is stated without caveats, but `grep tests/ _page_id` returns 3 existing hits that will break. Proposed clarification: add explicit sub-clause "Tests importing `_page_id` from `kb.utils.pages` (test_v0914_phase395.py line 732; test_v0915_task01.py lines 344, 352) must be migrated to the new public `page_id` in the same atomic commit, OR `_page_id = page_id` kept as a deprecated alias."
- F3: AC8 — "at minimum assert the short-circuit short-path does not import `kb.config`" is a weaker fallback than the stub-config path. Proposed clarification: pick one strategy as the primary contract; the other is not needed.
- F4: AC1 — "No site raises `AttributeError`" is the pass criterion. Implicit assumption: when `_coerce_str_field` raises `ValueError`, that's acceptable. Proposed clarification: restate as "No site raises `AttributeError` or `TypeError`; a clean `ValueError` from pre-validation is the expected failure surface."
- F5: AC2 — brainstorm OQ3 decides `ingest_source` RAISES `ValueError` (not return error dict). Requirements AC2 still offers both ("returns `{"error": "...", ...}` or raises `ValueError`"). Proposed clarification: strike the return-dict option; AC2 says "raises `ValueError` with the message naming `kb_create_page`."
- F6: AC6 — "scan_wiki_pages skips index sentinel files (`index.md`, `_sources.md`, `_categories.md`, `log.md`, `contradictions.md`, `purpose.md`, `hot.md`)" — `scan_wiki_pages` in the current code scans `*.md` only in `WIKI_SUBDIRS` (the 5 type subdirs) — index sentinels live at the wiki ROOT, so they're already never scanned. The test as written would either test a non-behavior (sentinels are excluded because root-level files are never enumerated) or would need to put sentinels inside a subdir to see them excluded. Proposed clarification: AC6 case (c) is a documentation test, not a behavior test — reframe as "`scan_wiki_pages` only returns files from `WIKI_SUBDIRS`; root-level files (index.md etc.) are not enumerated."

## Architectural / Contract / Integration Concerns

The main code path is architecturally conservative: no `src/` files changed, and the production snapshot semantics are correctly recognized. `kb.capture` snapshots both `atomic_text_write` and `call_llm_json` via `from ... import ...` at `src/kb/capture.py:36-37`, so tests that patch only the importer binding are vulnerable if `kb.capture` is deleted and reimported. AC1 and AC6 patch the canonical source first, then the importer binding, which is the right pattern for this style of module-top import.

The AC0 subprocess refactor is the stronger fix for the original contamination source. It removes the parent-process `sys.modules["kb.capture"]` mutation entirely and leaves the dual-site fixture as defense-in-depth. The local Windows environment could not execute the symlink RuntimeError path because symlink creation was skipped, so the POSIX RuntimeError probe is not dynamically verified here. Static review shows the guard is targeted precisely and CI's editable install should make child imports work.

AC5 is the only substantive contract gap. The implemented test proves "both owners are patched to the same fake" and would fail if the utils-side setattr is removed. It does not prove the fresh-module reimport behavior described in the top docstring. I do not consider this request-changes because AC0 eliminated the in-process deletion source, and the identity assertion is enough to pin AC1's dual-site fixture shape. Still, adding the reimport case or narrowing the doc wording would make the regression contract match the architecture story.

The BACKLOG cleanup is structurally sound: resolved entries are gone, unresolved residuals are re-pinned, and the new fold-into-canonical entry is present. The doc-update routing mostly follows the CLAUDE slim-index rule, with per-topic detail placed in `docs/reference/implementation-status.md` and `docs/reference/testing.md`; the stale duplicate cycle number in `CLAUDE.md` should be cleaned before merge.

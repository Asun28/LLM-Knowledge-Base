## Analysis

The cycle-24 surface fits a common "closing bounded gaps" shape: four clusters (evidence-trail collapse, vector-index atomic rebuild, lock-poll backoff, BACKLOG maintenance) plus a newly surfaced Cluster E (sentinel-anchor hardening) discovered by the Step-2 threat model. All five clusters have narrow blast radii that stay inside their declared files. The most structurally interesting aspect ŌĆö and the thing an R1 reviewer should explicitly surface ŌĆö is that the cycle 24 scope *grew by one cluster* between Step 1 (requirements) and Step 2 (threat model), and the new cluster (AC14/AC15) is non-optional. T5 in the threat model is not speculative: once AC1 moves the sentinel into the first-write payload, any attacker who can plant a sentinel-looking substring into `raw/articles/foo.md` body content can hijack every subsequent `append_evidence_trail` on the derived page, because `content.index(SENTINEL)` finds the attacker-planted bytes FIRST. This is genuinely a regression introduced by AC1, not a pre-existing vulnerability ŌĆö pre-cycle-24 pages didn't emit a sentinel at first write, so the `append_evidence_trail` FIRST-match heuristic only encountered the sentinel at the top of a real `## Evidence Trail` section. AC14 closes the newly-introduced hole with `## Evidence Trail` header anchoring. Both ACs must ship together or neither should ŌĆö a partial ship of AC1 without AC14 actively widens the attack surface.

The second structural observation concerns the `_write_wiki_page` exclusive branch. Cycle 20 AC8 added an `exclusive=True` code path that uses `os.open(O_WRONLY|O_CREAT|O_EXCL)` + POSIX `O_NOFOLLOW` + poison-unlink to defeat summary-slug collisions. That branch, at lines 358-390 of `pipeline.py`, currently writes the body via `os.write(fd, rendered.encode("utf-8"))` AND THEN calls `append_evidence_trail` OUTSIDE the open-fd span (at line 393). This is a two-write race just like the `exclusive=False` branch ŌĆö AC1 Condition-1 in the requirements explicitly says the inline render applies to BOTH branches. A reviewer's risk with this language is subtle: because the exclusive branch uses `os.write` + poison-unlink rather than `atomic_text_write`, the "inline render" must be concatenated into `rendered` BEFORE the `os.write(fd, rendered.encode(...))` call. Any implementation that only adds the trail concatenation to the `atomic_text_write` branch leaves the exclusive branch with the original two-write race ŌĆö which is a hidden regression that wouldn't fire in the tmp-project fixtures unless the test explicitly uses `exclusive=True` (two of three summary/item call sites at lines 1056, 1419 pass `exclusive=True`). Conclusion: AC1 test coverage must parameterize over `exclusive Ōłł {False, True}` OR explicitly name both branches in separate tests, and the requirements text at Condition 1 already pins this.

The third structural observation concerns `VectorIndex.build`'s empty-entries fast path (lines 304-309). That branch opens `sqlite3.connect(str(self.db_path))` unconditionally; AC7's `db_path` kwarg migration MUST route both the empty and the populated path through the same override logic. The current code uses `self.db_path` in two distinct code locations (line 306 for empty, line 314 for populated), so AC7's implementation sketch in brainstorm needs to be explicit about updating BOTH. Concretely, `db_path = db_path if db_path is not None else self.db_path` must bind once at function entry (per T10 recommendation), and subsequent `sqlite3.connect(str(db_path))` calls use that local. The brainstorm sketch at V1 step 1 is correct; the R1 review flags that the implementer must not optimize by branching early on empty-entries (a natural "return after `sqlite3.connect; close`" simplification) and leaving the `db_path` override un-threaded into the empty branch. Tests T10 in threat model pins the `is not None` semantic ŌĆö the empty path must honor the same override.

The fourth observation concerns `_index_cache.pop` ordering (Condition 2). The existing code at lines 121-122 (empty-pages branch) and lines 134-135 (populated-pages branch) runs the pop AFTER the `VectorIndex(vec_path).build(...)` call. After the AC5 refactor, the pop must move BEFORE the `os.replace(tmp_path, vec_path)` call to release any Windows file handle held by a cached `VectorIndex._conn` before NTFS kernel-locked the target of the rename. On POSIX the ordering doesn't matter for correctness ŌĆö the OS handles rename-over-open-fd fine ŌĆö but the Condition-2 language in requirements pins the ordering for cross-platform consistency. The empty-pages branch needs the same care: it currently constructs a fresh `VectorIndex(vec_path).build([])` which creates a new DB file, then pops. Under the refactor, the build goes to tmp, pop runs, replace happens. Code-review checkpoint: both branches (empty + populated) need the four-line tmp-path pattern, not just the populated one.

## T1 — Exception `__str__` leaks filesystem paths to logs before MCP redaction

- **Severity:** MEDIUM
- **Surface:** `kb.errors.StorageError.__str__`, `kb.errors.IngestError.__str__` — any library-level `logger.exception(...)` call before the MCP envelope.
- **Scenario:** AC8 shapes `StorageError(f"summary_collision: {path}", kind="summary_collision")`. If a caller logs via `logger.exception("ingest failed")` before the exception bubbles to `kb.mcp.quality._sanitize_error_str`, the absolute path of the project lands in server logs. Remote log aggregators (container platforms) then ingest the path, disclosing deployment layout. Same risk for `CompileError(str(e))` wrapping an `OSError(2, "No such file or directory: 'C:\\Users\\Admin\\...'")`.
- **Mitigation:** `StorageError.__init__` takes a separate `path: Path | None = None` kwarg stored on the instance but NOT rendered in `__str__`; `__str__` returns only `f"{kind}: <path_hidden>"` when `kind` is present and `path` is set, or `<msg>` otherwise. MCP wrappers (AC14, AC17) continue to route through `_sanitize_error_str` which handles the belt-and-braces case. AC5 (narrow hot-path migration) must keep the existing `from e` chaining so `__cause__` carries the full OSError for local debug without forcing it into `str(err)`. ACs: AC1, AC5, AC14.
- **Verification:** Codex greps (1) `rg -n "raise StorageError\(f?\".*\{.*path" src/` returns zero hits where the path is interpolated into the message string without redaction; (2) `rg -n "logger\.(exception|error).*StorageError|IngestError|CompileError" src/` inspected for path leakage; (3) `_sanitize_error_str` test covers `StorageError(kind="summary_collision")` input.

## T2 — `isinstance(e, Exception)` regressions from reparenting `LLMError` / `CaptureError`

- **Severity:** LOW
- **Surface:** `kb.utils.llm.LLMError`, `kb.capture.CaptureError` + all existing `except LLMError:` / `except Exception:` sites.
- **Scenario:** AC2 reparents both to `KBError(Exception)`. The MRO keeps `isinstance(err, Exception)` True (Exception is still an ancestor), so library call sites are safe. The risk is a test that does `assert type(err) is Exception` or `assert err.__class__.__bases__ == (Exception,)` — behaviourally silly, but such a test would now fail.
- **Mitigation:** AC6 regression test explicitly asserts `issubclass(LLMError, KBError)` AND `issubclass(LLMError, Exception)` — the latter guards against future re-parenting accidentally breaking the outer catch. Pre-migration grep for `type\(.*\) is Exception|__bases__` in tests/ (expect zero). ACs: AC2, AC6.
- **Verification:** Codex greps `rg -n "type\(.*(LLMError|CaptureError).*\) is Exception" tests/` returns zero hits; AC6 test file imports and asserts MRO relations.

## T3 — O_EXCL double-race: evidence-trail read-before-write hole under concurrent ingest

- **Severity:** HIGH
- **Surface:** `kb.ingest.pipeline._write_wiki_page(exclusive=True)` called from summary write (AC9) and item batch write (AC10).
- **Scenario:** Two concurrent `ingest_source` calls with colliding slugs both see `path.exists() == False` (the legacy TOCTOU check), both call `_write_wiki_page(..., exclusive=True)`. Thread A wins `os.open(O_EXCL)`, Thread B gets `FileExistsError` → `StorageError(kind="summary_collision")`. AC9/AC10 say "pivot to `_update_existing_page`." But B's `_update_existing_page` now races against A's *evidence-trail append* (A's `append_evidence_trail` may not have completed when B starts reading). Under bad ordering B reads the freshly-created page with A's frontmatter present but A's evidence-trail still pending — B's merge then appends to a half-written file. Symlink-target swap between `open` call and `write` call is a secondary concern on POSIX (Windows O_EXCL blocks named-symlink replacement but not junction swap).
- **Mitigation (as shipped — updated from the original draft per Step 8 plan-gate amendment):** O_EXCL is the atomic body-reserve primitive. `_write_wiki_page` deliberately does NOT acquire `file_lock(page_path)` at the caller or internally around the create+evidence-trail pair because `append_evidence_trail` **already acquires `file_lock(page_path)` internally** (cycle-1 H1 contract, `src/kb/ingest/evidence.py:81`) and the sidecar lock is NOT re-entrant — a caller-held wrapper would self-deadlock. Serialisation of the A-vs-B race therefore happens via TWO separate lock acquisitions against the same `.lock` sidecar: (a) Thread A's `append_evidence_trail` acquires and releases, and (b) Thread B's `_update_existing_page` acquires (cycle-20 AC11/D-NEW-1 — unconditional `file_lock(page_path)` now inside the helper body, `src/kb/ingest/pipeline.py _update_existing_page_body`) and releases after the merge write. Because `file_lock` uses `os.O_CREAT|O_EXCL` on the sidecar, the two acquires are mutually exclusive — Thread B blocks until Thread A's evidence append completes, so B's read sees the fully-formed page. Thread B's `_update_existing_page_body` then releases its lock BEFORE calling `append_evidence_trail` (to avoid the same non-re-entrancy problem), and the final evidence trail acquires/releases independently. `os.open` flags include `O_NOFOLLOW` on POSIX (`os.O_NOFOLLOW` where available, `hasattr(os, "O_NOFOLLOW")` guarded) to defeat symlink-target swap; on Windows CPython's `CreateFileW(CREATE_NEW)` already refuses existing reparse points. Umask: `0o644` explicit mode sets a predictable permission floor; Windows ignores the mode but NTFS inherits ACL from parent. File-creation via `_write_wiki_page` only targets `wiki/{summaries,entities,concepts,comparisons,synthesis}/` — all project-owned directories. AC12 thread-race test drives real `ingest_source` calls in 2 threads with `threading.Barrier(2)` and asserts BOTH sources land in the final frontmatter `source:` list AND BOTH evidence-trail entries survive. ACs: AC8, AC9, AC10, AC11, AC12.
- **Verification:** Codex greps (1) `rg -n "O_EXCL\|O_NOFOLLOW\|summary_collision" src/kb/ingest/pipeline.py` returns hits covering the O_EXCL reserve + POSIX guard + StorageError kind; (2) `rg -n "with file_lock" src/kb/ingest/pipeline.py` shows the unconditional lock inside `_update_existing_page_body` AND confirms no caller-held wrapper around `_write_wiki_page(exclusive=True)` (that wrapper would deadlock); (3) `rg -n "append_evidence_trail" src/kb/ingest/pipeline.py` shows calls OUTSIDE the `with file_lock(...)` block (evidence trail has its own sidecar lock per `src/kb/ingest/evidence.py:81`); (4) `tests/test_cycle20_write_wiki_page_exclusive.py::test_concurrent_ingest_merges_both_sources_and_evidence_trails` drives the race and asserts both sources + both evidence entries.

## T4 — `sweep_stale_pending(action="delete")` is irreversible and audit-free

- **Severity:** MEDIUM
- **Surface:** `kb.review.refiner.sweep_stale_pending`, MCP `kb_refine_sweep`, CLI `kb refine-sweep`.
- **Scenario:** Operator runs `kb refine-sweep --age-hours 1 --action delete` (typo for 168) → all pending rows whose write happened within the last hour are permanently removed from `review_history.json`. The original page_ids, revision_notes, and attempt_ids are gone; no before-state snapshot is written. A DoS variant: attacker who can reach the MCP boundary (local-only per `.mcp.json`, but still: any shell on the host) passes `hours=0` (→ rejected by AC13 `hours < 1`) or `hours=1` to silently clear recent pending rows and hide evidence of a failed refine race. File-lock starvation: extremely long history files with many pending rows hold `file_lock(history_path)` for the load → mutate → save span, blocking concurrent `refine_page` (which locks `history_path` SECOND per cycle-19 AC9 lock-order). Because refine_page holds page_lock FIRST, a long sweep does NOT block page reads, only refines — acceptable.
- **Mitigation:** AC13 enforces `action in {"mark_failed", "delete"}` with `mark_failed` as the safe default. `delete` MUST write a one-line summary to `wiki/log.md` via `append_wiki_log("sweep", f"Deleted {N} stale pending rows; action=delete; sweep_id={id}; cutoff={cutoff.isoformat()}")` — creates an audit trail outside the mutated file. `mark_failed` default + `sweep_id` + `sweep_at` + `error="abandoned-by-sweep"` tag preserves the row content, making `mark_failed` fully reversible. `hours` validation `>= 1` blocks the zero-hour DoS. Lock span stays single (no release/reacquire) per cycle-19 AC9 — concurrent refine_page serialises cleanly. ACs: AC13, AC14, AC15, AC16.
- **Verification:** Codex greps (1) `rg -n "action == \"delete\"" src/kb/review/refiner.py` shows the `append_wiki_log` audit-trail call within the delete branch; (2) AC16 test includes "delete writes wiki/log.md entry" assertion; (3) no path allows `action` values outside the enumerated set.

## T5 — `kb_refine_list_stale` / `kb_refine_sweep` dry_run disclose `revision_notes` content to remote MCP callers

- **Severity:** LOW
- **Surface:** MCP `kb_refine_list_stale` response payload AND MCP `kb_refine_sweep(dry_run=True)` candidates list (extension added after PR #34 R3 Sonnet review).
- **Scenario:** Operator-supplied `revision_notes` in `kb_refine_page` calls may contain internal context ("fixing CVE-2025-XXXXX per issue #42"). `list_stale_pending` returns full dict copies including `revision_notes`. AC17 as drafted returns every field. A future MCP deployment exposed to non-operators would leak this.
- **Mitigation:** AC17 projects a minimal field set: `[{"attempt_id", "page_id", "timestamp", "notes_length"}]` — omits `revision_notes`. The cycle-20 R3 Sonnet review (PR #34) surfaced that `kb_refine_sweep(dry_run=True)` ALSO carries a `candidates` list of full-row dicts that reaches the MCP boundary via `json.dumps(result)`; the mitigation now extends to `kb_refine_sweep` — when `result["dry_run"]` is True, each candidate row in the MCP wrapper is projected to the same `{attempt_id, page_id, timestamp, notes_length}` field set. Library helpers (`list_stale_pending`, `sweep_stale_pending`) keep full dicts for CLI/local use; only the MCP wrappers filter. Document the projection in both MCP tool docstrings. ACs: AC17, AC19 (+ R3-fix for `kb_refine_sweep`).
- **Verification:** `rg -n "revision_notes" src/kb/mcp/quality.py` returns zero hits in either MCP wrapper; the AC19 tests + new `test_dry_run_candidates_omit_revision_notes` assert returned JSON does NOT contain `revision_notes` for both surfaces.

## T6 — Windows tilde-path test: `ctypes` buffer overflow / vacuous skip

- **Severity:** LOW
- **Surface:** `tests/test_cycle20_windows_tilde_path.py::test_canonical_rel_path_tilde_equivalence`.
- **Scenario:** `GetShortPathNameW` writes into a caller-allocated buffer; CPython `ctypes` call must pass a `create_unicode_buffer(260)` sized correctly or the short-name write overflows. A malformed call returning a garbage path would then be fed to `_canonical_rel_path`, which calls `Path.resolve()` — exceptions (not path injection) since `_canonical_rel_path` already has the cycle-19 null-byte + traversal validator. No persistence risk (tmp_path isolation). Residual risk: if the filesystem has 8.3 disabled AND the test forgets to skip, `GetShortPathNameW` returns the long form verbatim and the equivalence assert is vacuous.
- **Mitigation:** AC20 specifies `pytest.skip` WITHIN the test when short-form == long-form (filesystem has 8.3 disabled). `ctypes` call uses `ctypes.create_unicode_buffer(260)` with explicit bounds. `@pytest.mark.skipif(sys.platform != "win32")` gates platform. Test uses `tmp_path` fixture — automatic teardown. ACs: AC20.
- **Verification:** Codex greps (1) `rg -n "create_unicode_buffer" tests/test_cycle20_windows_tilde_path.py` shows sized buffer; (2) test has BOTH platform skipif AND runtime skip; (3) no `os.environ` or global-state mutation.

## T7 — diskcache `CVE-2025-69872` is class-D (indirect) but status check must run

- **Severity:** LOW (exposure) / MEDIUM (hygiene)
- **Surface:** `requirements.txt` pin of `diskcache==5.6.3` (transitive via trafilatura/crawl4ai robots.txt caching).
- **Scenario:** Zero direct imports of `diskcache` exist in `src/kb/`. The CVE requires an attacker to control a pickle-serialized cache entry — reachable only if an upstream library we call feeds attacker-controlled bytes into a diskcache `pickle.loads` path. Current reachable paths: trafilatura robots.txt caching (attacker-controlled if we fetch an untrusted host's robots.txt, which `kb.lint.fetcher` DOES via `SafeTransport` — but `SafeTransport` does not enable pickle caching and the content fetched is parsed as plain text, not pickled). So the class-D exposure is "no known kb-originated reachable sink."
- **Mitigation:** AC21 runs `pip index versions diskcache` and compares against GHSA `first_patched_version`. If patched upstream: bump pin + CHANGELOG + delete BACKLOG entry. If still unpatched: dated `# Re-checked 2026-04-21` note in BACKLOG + CHANGELOG. Visibility is the deliverable. Step 11.5 CVE-patch pass SKIPS since there is no patch yet; Step 15 late-arrival watcher keeps the alert pinned. ACs: AC21.
- **Verification:** Codex (1) Dependabot-alerts check against cycle-19 baseline shows zero new alerts; (2) BACKLOG entry has a dated re-check note OR is deleted with CHANGELOG bump-entry present; (3) `rg -n "diskcache|DiskCache|FanoutCache" src/kb/` returns zero direct imports.

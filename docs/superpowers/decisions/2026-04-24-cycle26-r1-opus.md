## Per-AC scoring

### AC1 — `maybe_warm_load_vector_model(wiki_dir)` helper

- **Framing:** testable (4 explicit None-return conditions + thread-return + log assertion). **Over-specifies the return type** (`threading.Thread | None`) — that's fine for the AC but do not let it leak into public docs unless we intend to keep the `threading.Thread` as a public type forever.
- **Semantic correctness:** three guard clauses resolve against symbols that EXIST (lines 21, 86, 31). "`_model is not None` → idempotent None" is the correct short-circuit.
- **Risk:** **Idempotency race hole.** The AC specifies a lock-free `if _model is not None: return None` check. Two concurrent callers both observing `_model is None` will each spawn a daemon thread; the second thread blocks on `_model_lock` inside `_get_model()` and returns immediately after the first completes — but we've spent an extra thread spawn + stack frame. T5 acknowledges this; production sees exactly ONE caller (`main()`), so impact is bounded. **Acceptable** — but document the asymmetry in the docstring (the AC2 single-caller invariant is what keeps T5 low-likelihood, NOT the helper's self-protection).
- **Verdict: AMEND-SCOPE.** Add a single clarifying line to the AC: "The helper does NOT guarantee single-spawn under concurrent callers (T5 accepted); production callers must be single-site per AC2." Avoids future maintainers adding a spurious `_warm_load_lock`.

### AC2 — Wire warm-load hook into `kb.mcp.__init__.main()`

- **Framing:** testable (import-ordering assertion via subprocess probe; `threading.Thread` mock assertion).
- **Semantic correctness:** `main()` exists at line 57 with `_register_all_tools()` at line 61, `_mcp = from kb.mcp.app import mcp` at line 62, `_mcp.run()` at line 66. Insertion slot is clean (line 65-66 is the logging-setup then run). AC says "AFTER `_register_all_tools()` and BEFORE `_mcp.run()`" — matches reality.
- **Risk:** **Function-local import contract is UNDERSPECIFIED.** The AC says "Import must be function-local (preserves cycle-23 AC4 boot-lean contract — bare `import kb.mcp` must not load embeddings module)." But the existing cycle-23 boot-lean test does NOT include `kb.query.embeddings` in its heavy-deps list. A naïve implementation that adds `from kb.query.embeddings import maybe_warm_load_vector_model` at module scope of `kb/mcp/__init__.py` would pass all existing tests and silently break the intended boot-lean contract. Applies threat: this is NOT T1-T6 — it's a **pre-existing test-coverage gap** the cycle creates a new regression vector for.
- **Verdict: AMEND-SCOPE.** Cycle 26 MUST extend `tests/test_cycle23_mcp_boot_lean.py::test_bare_import_kb_mcp_does_not_pull_heavy_deps` to ADD `'kb.query.embeddings'` to the heavy-deps allowlist. Without this test extension, AC2's "preserves cycle-23 AC4" claim is unenforceable. Add this as AC2b OR as an explicit test requirement inside AC5.

### AC3 — Cold-load latency instrumentation

- **Framing:** testable (monkeypatch `StaticModel.from_pretrained` with sleeper; assert INFO + WARNING via caplog).
- **Semantic correctness:** `_get_model()` is the correct site. `time.perf_counter` is the right clock (monotonic, doesn't drift). Threshold constant `VECTOR_COLD_LOAD_WARN_THRESHOLD_SECS = 0.3` matches backlog spec.
- **Risk:** **T1 path leak** — none (AC3 log lines don't include paths). **Test-stability risk** — a caplog test that asserts BOTH info+warning fired in order needs a deterministic sleep; AC5 stub at 0.5s vs 0.3s threshold has healthy margin. **What if `StaticModel.from_pretrained` raises?** The instrumentation must measure elapsed only on success; failure path should NOT log "cold-loaded" (misleading). AC3 says "On success:" — correct framing. Implementation MUST place the `logger.info/warning` AFTER the `_model = StaticModel.from_pretrained(...)` assignment, NOT in a `finally:`.
- **Verdict: APPROVE.** Add implementation hint: "timing span wraps only the `StaticModel.from_pretrained` call; log fires AFTER successful assignment to `_model`, NOT in finally." (Prevents the misleading-log-on-failure footgun.)

### AC4 — Cold-load counter `_vector_model_cold_loads_seen` + `get_vector_model_cold_load_count()`

- **Framing:** testable via `_reset_model()` + `_get_model()` cycles.
- **Semantic correctness:** mirrors cycle-25 `_dim_mismatches_seen` (line 52) + `get_dim_mismatch_count()` (line 55). Variable naming is consistent.
- **Risk:** **T4 asymmetry (counter inside vs outside lock)** — cycle-25 counter is lock-free (Q8); cycle-26 proposes locked (Q4). Defensible on cold-load cost grounds (load is ~0.8s, the lock is already held, exact counts are free). However: **the counter must increment AFTER the successful `_model = ...` assignment, NOT on every entry into the critical section** — otherwise a `from_pretrained` exception would still bump the counter ("cold-load succeeded" lie). Also: `get_vector_model_cold_load_count()` should NOT take the lock on read — a stale read is fine for diagnostic counters, and adding lock-on-read defeats the "cheap observability" premise.
- **Verdict: APPROVE.** Pin three invariants in the implementation: (a) increment AFTER `_model = StaticModel.from_pretrained(...)` succeeds, (b) increment INSIDE `_model_lock` (T4 Q4 decision), (c) `get_vector_model_cold_load_count()` reads lock-free (consistent with cycle-25 getter pattern).

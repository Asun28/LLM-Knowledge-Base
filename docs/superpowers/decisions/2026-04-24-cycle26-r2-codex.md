- The main concurrency risk is duplicate warm-load threads, not duplicate model loads: `_get_model()` already uses double-checked locking (`src/kb/query/embeddings.py:270-279`), but AC1 explicitly adds a pre-spawn `_model is not None` check outside that lock (`2026-04-24-cycle26-requirements.md:35-39`), so two processes or two callers can still race to start redundant daemon threads.
- AC5’s logging test should set capture level explicitly. The AC asks to assert an `info` log via `caplog` (`2026-04-24-cycle26-requirements.md:54-66`), while current server boot config sets the root logger to `WARNING` when it owns handler setup (`src/kb/mcp/__init__.py:64-65`).
- Warm-load failure retry semantics are acceptable if the wrapper only logs and leaves `_model` unset. Current `_get_model()` assigns `_model` only after `StaticModel.from_pretrained(...)` returns (`src/kb/query/embeddings.py:275-278`), so a failed warm-load naturally re-attempts on the next `_get_model()` call.
- The BACKLOG deletion claim is overstated as written: the requested grep for `multiprocessing\.Process` returned no matches, while `@pytest.mark.integration` is present and the file itself shows `ctx.Process(...)`, Event handshake, and PID sentinel assertions (`tests/test_cycle23_file_lock_multiprocessing.py:64-91`).
- Counter asymmetry is defensible. Cycle 25’s `_dim_mismatches_seen` is explicitly lock-free and approximate on the query path (`src/kb/query/embeddings.py:46-68`); cold-load counting belongs inside `_model_lock` because that lock is already mandatory for singleton creation (`src/kb/query/embeddings.py:31-32`, `270-279`).

1. Concurrency edge cases

AC1/AC2 add warm-load before `_mcp.run()` (`2026-04-24-cycle26-requirements.md:43-45`), and the threat model assumes FastMCP workers may later call `_get_model()` concurrently (`2026-04-24-cycle26-threat-model.md:27-45`, `102-112`). Within one process, duplicate loads are prevented by `_model_lock` (`src/kb/query/embeddings.py:273-279`). Hypothesis: duplicate daemon threads are still possible if callers race between the helper’s outer `_model is not None` guard and `Thread.start()`. That is wasteful but not correctness-breaking. Q6’s bias to swallow `Thread.start()` `RuntimeError` and continue MCP boot is correct because current `main()` has no model dependency before `_mcp.run()` (`src/kb/mcp/__init__.py:57-66`; `2026-04-24-cycle26-requirements.md:85`). `_hybrid_available` is computed once at import (`src/kb/query/embeddings.py:17-25`); a `True→False` flip requires test-driven module reload hypothesis, not ordinary runtime behavior. Reload would also recreate `_model` and `_model_lock` (`src/kb/query/embeddings.py:31-32`), so tests should avoid mixing pre/post-reload references.

2. Test reliability

I would amend AC5 to require `caplog.set_level(logging.INFO)` or equivalent logger-targeted capture. The reviewed files do not show any existing pytest logging override, and the only explicit logging baseline in reviewed code is `basicConfig(level=logging.WARNING, ...)` in `main()` (`src/kb/mcp/__init__.py:64-65`). Since AC5 specifically asserts the info message (`2026-04-24-cycle26-requirements.md:54-66`), relying on default capture is brittle.

3. Model load semantics

Observed behavior: `_model` remains `None` until `StaticModel.from_pretrained(...)` returns successfully (`src/kb/query/embeddings.py:275-278`). Therefore a warm-load exception does not poison the singleton; the next query re-enters `_get_model()` and retries. That matches the brainstorm/threat-model recommendation to log and let the first real query retry (`2026-04-24-cycle26-brainstorm.md:112-114`; `2026-04-24-cycle26-threat-model.md:158-178`). Hypothesis: if upstream HF-Hub performs its own internal retries, total wait may be long, but I do not see any loop in local code that would create an unbounded retry storm beyond “one attempt per caller.”

4. Backlog deletion correctness

The exact grep claim in AC6 is not fully satisfied. `rg 'multiprocessing\.Process' tests/test_cycle23_file_lock_multiprocessing.py` returned no results. `rg 'pytest\.mark\.integration' ...` did match. Still, the file itself shows equivalent evidence: `import multiprocessing as mp` (`:22`), `ctx.Process(...)` spawn (`:64-76`), Event-based handshake (`:65-66`, `82-85`), PID sentinel assertion (`:87-91`), and the integration marker (`:55`). Conclusion: the stale BACKLOG item is substantively closed, but AC6 should cite `ctx.Process(` or `mp.get_context("spawn")` rather than the stricter absent pattern.

5. Counter asymmetry

Current proposal is correct. The dim-mismatch counter is hot-path, lock-free, and documented as approximate (`src/kb/query/embeddings.py:46-68`). Cold-load is already serialized by `_model_lock` (`270-279`), so incrementing `_vector_model_cold_loads_seen` there is exact at near-zero added cost. I would document the asymmetry explicitly to reduce maintenance confusion, but I would not force consistency for its own sake.

6. Missed ACs

Yes: AC3 omits sibling observability for first-query total latency. The requirements define the user problem as “operators cannot distinguish ‘MCP slow’ from ‘vector model loading’” (`2026-04-24-cycle26-requirements.md:11-12`), but the reviewed code path shows `_get_model()` is only one contributor. The cycle should at least note that index-open / query-engine startup latency remains uninstrumented, even if it stays out of scope this round.

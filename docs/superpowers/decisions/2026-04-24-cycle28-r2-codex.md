# Cycle 28 — R2 Codex Design Eval

## Edge-case findings
1. Finding: `_ensure_conn` already uses the same double-checked pattern as `_get_model`: lock-free fast path, then `_conn_lock`, then a second `_conn is not None` check before side effects. If cycle 28 increments `_sqlite_vec_loads_seen` only on the post-success path inside `_conn_lock`, the count is exact per successful first connect and matches cycle-26 AC4 style. Severity: low. Fix: increment only after `sqlite_vec.load(conn)` succeeds and before returning, never in `finally:`.
2. Finding: `engine.search_pages()` releases `_WIKI_BM25_CACHE_LOCK` before `BM25Index(documents)`, and the raw-source path does the same with `_RAW_BM25_CACHE_LOCK`. Two concurrent misses can therefore both build and both increment `_bm25_builds_seen`. That is acceptable only if the counter is explicitly documented as approximate diagnostic telemetry. It is not suitable for exact “number of cache rebuilds” assertions. Severity: medium. Fix: keep the counter lock-free, but pin tests to monotonic deltas and document undercount/overcount-relative-to-cache-win semantics.
3. Finding: On Python 3.12, `time.perf_counter()` is the correct clock for elapsed-duration timing; Python documents it as a high-resolution performance counter, and `get_clock_info("perf_counter").monotonic` should be true. Negative elapsed from real clock rollback is not a credible Windows risk here. Float rollover is also not practical. Severity: low. Fix: none for production; optionally assert monotonicity via `time.get_clock_info("perf_counter")` in a unit test if the team wants belt-and-suspenders. Source: https://docs.python.org/3.12/library/time.html
4. Finding: The reliable pytest pattern is fixture-based monkeypatching, but the better target is the module-local lookup, e.g. `monkeypatch.setattr("kb.query.embeddings.time.perf_counter", fake)` after the module imports `time`, or `monkeypatch.setattr(bm25_mod.time, "perf_counter", fake)`. Patching global `time.perf_counter` is broader and easier to leak conceptually into unrelated code under test, even though pytest will revert it. Cycle-26’s precedent file does not actually patch `perf_counter`; it uses a slow stub instead. Severity: medium. Fix: write cycle-28 tests with module-scoped monkeypatch targets, not raw assignment and not repo-global `time.perf_counter` unless intentionally broad.
5. Finding: No current test directly reloads `kb.query.embeddings`, but the suite does reload `kb.config` in several places and has existing commentary about reload drift across sibling modules. So T8 is real as a class of risk even if there is no direct grep hit for `reload(...embeddings...)`. Severity: medium. Fix: keep getters read-only and make all counter tests baseline-relative; do not add tests that assert absolute counter values from process start.
6. Finding: `BM25Index.__init__` sets `self.n_docs = len(documents)` before any future log emission point. Empty corpus already lands at `self.n_docs == 0` and `avgdl = 1.0`, so `n_docs=0` will render correctly if the INFO log is emitted at the end. Severity: low. Fix: none beyond keeping the log after `self.n_docs` initialization.
7. Finding: Cycle 28 does not threaten the boot-lean contract as long as it only adds standard-library imports inside `bm25.py` / `embeddings.py`. The boot-lean tests target bare `import kb.mcp`, and `kb.mcp.__init__` still uses function-local imports for embeddings. Import cost inside `bm25.py` or `embeddings.py` matters only if those modules become newly imported from `kb.mcp` at package import time, which this cycle should not do. Severity: low. Fix: do not add any new `kb.query.embeddings` or `kb.query.bm25` import to `kb.mcp.__init__` module scope.
8. Finding: `pytest-xdist` is not enabled in `pyproject.toml`, but cycle-25 design notes explicitly treated monotonic-delta counters as xdist-safe because workers are per-process. No current infra assumes cross-worker exact counts. Severity: low. Fix: none in production; keep tests phrased as local-process deltas, not suite-global totals.
9. Finding: A single `_bm25_builds_seen` counter can cover both `engine.py:110` wiki-page BM25 builds and `engine.py:794` raw-source BM25 builds if the intent is “how many BM25 indices did this process build?” It cannot answer “which cache family was expensive?” and therefore slightly muddies first-query diagnosis when wiki and raw searches coexist. Cycle-4 history deliberately separated `_WIKI_BM25_CACHE` and `_RAW_BM25_CACHE`, so the caches are semantically distinct even if the class is shared. Severity: medium. Fix: minimum acceptable change is to state in the getter docstring that the counter aggregates both call sites; better fix is separate counters if the operator question is per-cache-family.
10. Finding: For logfmt-style downstream parsing, key presence matters more than word order, but cycle-25/26 local convention is “human text first, then structured-ish parenthetical key=value tail” such as `(... vec_db=%s)` or `(threshold=...)`. `sqlite-vec extension loaded in %.3fs (db=%s)` and `BM25 index built in %.3fs (n_docs=%d)` match that convention. Reversing to `db=%s in %.3fs` is not necessary and is less consistent with existing logs. Severity: low. Fix: preserve the proposed cycle-28 format; if the project wants machine-friendly logs, solve that with structured logging later, not ad hoc word-order changes.

## Integration risks
- Upstream callers do not change behavior, but test writers may misread `_bm25_builds_seen` as “cache rebuild count” rather than “constructor executions,” especially because duplicate concurrent builds can be discarded after the cache winner check.
- Downstream observability consumers get only process-local counters. They reset on interpreter restart and can also reset on module reload in tests.
- The raw and wiki BM25 caches remain separate while the proposed counter is shared; that is fine for coarse telemetry, not for source-specific attribution.
- Boot-lean safety depends on keeping all new imports out of `kb.mcp` package-import scope.

## Security findings
- Threat-model coverage is broadly sufficient. The remaining notable exposure is that AC1 logs an absolute DB path and AC4 logs corpus size; both are existing-pattern low-risk disclosures, not new trust-boundary breaks.
- The main correctness/security-adjacent footgun is a future `finally:` around timing/logging/counter increments. That would emit success telemetry on failure paths and should stay explicitly prohibited.
- Keep getters non-MCP, non-CLI. Exposing them externally would turn approximate counters into operator-facing API and raise expectation mismatches.

## Perf findings
- Instrumentation overhead is negligible relative to the measured operations: two `perf_counter()` calls, one integer increment, and one INFO log on cold paths only.
- BM25 logging is unconditional per constructor call, so duplicate concurrent cache-miss builds also duplicate the INFO line; still acceptable because construction itself is already expensive.
- If the team wants to minimize overhead further, the only meaningful target is log volume, not the timer calls.

## Additional open questions not in brainstorm
- Q9. Is the operator question process-wide “how many BM25 builds happened?” or cache-family-specific “did wiki-page search rebuild vs raw-source search rebuild?” That answer determines whether one counter is sufficient.
- Q10. Should the cycle-28 test file also pin the failure-path invariant for `_ensure_conn` the way cycle-26 added a divergent-fail test for `_get_model`, to guard against a future `finally:` regression?
- Q11. Should the getter docstring for BM25 explicitly say “constructor executions, not distinct cache insertions” to prevent future test mis-specification?

## Overall verdict
APPROVE-WITH-CONDITIONS

Conditions: document `_bm25_builds_seen` as approximate constructor-count telemetry covering both wiki and raw call sites; write baseline-relative tests only; and patch `perf_counter` at the module-under-test scope so the new tests stay isolated and non-flaky.

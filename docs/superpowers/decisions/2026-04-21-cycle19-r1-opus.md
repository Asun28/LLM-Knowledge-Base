## Analysis

### AC1 — `inject_wikilinks_batch` signature
The tuple-list return type `dict[str, list[str]]` keyed by `target_page_id_lower` is structurally sound and mirrors the existing `build_backlinks(pages=)` contract at `linker.py:117`. Verified: `inject_wikilinks` at `linker.py:172` takes `(title, target_page_id, wiki_dir=None)` and returns `list[str]`; the batch extension is compositionally clean.

One latent issue: AC1 lists T-1 as "a wiki containing pages mentioning two new titles A and B receives wikilinks to both after one batch call; return dict has both keys", but AC3 explicitly guarantees **at most one** wikilink per page per batch. If a single existing page mentions both A and B, AC1's T-1 expectation of "both keys in the return dict" is only guaranteed when A and B hit **different** existing pages. The requirements should clarify that T-1 sets up pages-with-disjoint-mentions to avoid a false-positive PASS that is actually exercising AC3 logic by accident. Minor, fixable by test design.

**Verdict:** PASS

### AC2 — single-pass read amplification
The ceiling "`read_text` calls ≤ 10 (ignoring frontmatter cache reads)" conflicts with what the cycle-18 code actually does. `inject_wikilinks` currently performs **two** `read_text` calls per matched page: the pre-lock peek at `linker.py:222` and the under-lock re-read at `:244`. The open question Q2 in brainstorm.md acknowledges this and proposes "pre-lock peek + under-lock re-read = 2 per modified page; 1 per unmodified page", but AC2's T-2 copy still reads "≤ 10" without caveat.

For `len(new_pages)=5` and `len(existing_pages)=10`, the realistic budget is:
- Unmodified pages (no match): 1 read (peek only) × `U` = `U` reads
- Modified pages: 2 reads (peek + under-lock) × `M` = `2M` reads
- Total: `U + 2M = (10 - M) + 2M = 10 + M`

If all 10 existing pages have a candidate match (`M=10`), the budget is 20, not 10. The acceptance criterion needs to be rewritten as `≤ U + 2M ≤ 2·len(existing_pages)` or the test needs to construct pages that definitely don't match. Because the brainstorm already flagged Q2 as unresolved, this is fixable but material.

**Verdict:** MAJOR-budget-undercount

### AC3 — at-most-one wikilink per page
Tie-break rule "longest title first, alphabetical on pid" matches `_sort_new_pages_by_title_length` at `pipeline.py:153-163` (`sorted(pairs, key=lambda pt: (-len(pt[1]), pt[0]))`). The batch helper reusing this ordering is correct. The single-injection semantics per page match the existing single-target behaviour.

**Verdict:** PASS

### AC4 — `MAX_INJECT_TITLES_PER_BATCH = 200`
The constant is **not yet defined** in `config.py` (grep confirmed `MAX_NOTES_LEN`, `MAX_REVIEW_HISTORY_ENTRIES`, and other MAX_* constants but no `MAX_INJECT_TITLES_PER_BATCH`). Cycle 19 will add it — no blocker there.

The deeper question: 200 titles × (average title ~25 chars, per the `\b`-anchored regex with `re.IGNORECASE`) produces an alternation regex of roughly 5K chars. `re.compile` on this is O(n) in the alternation width with no catastrophic backtracking because every title is `re.escape`d (literal). The concern in T1 about ReDoS is really about compile cost + linear-scan cost, not worst-case backtracking. 200 is a reasonable ceiling; 500 would also be safe. The value is arbitrary but defensible.

**Verdict:** PASS

### AC5 — no-lock fast path preserved
`compile/linker.py:218-232` establishes the pattern: peek → frontmatter-strip → already-linked check → masked-body search → if no match continue (no lock). The batch helper extends this by running the combined-alternation scan against the masked body once per page; pages with zero matches take zero locks. The cycle-18 T8 invariant is preserved structurally.

**Verdict:** PASS

### AC6 — pipeline switches to batch call
The target site is `ingest/pipeline.py:1345-1353`. The existing loop calls `inject_wikilinks(ptitle, pid, wiki_dir=effective_wiki_dir)` per-title. The `except Exception` swallows any per-title failure with a DEBUG log (not WARNING) — design decision preserved.

One concern: the existing `except Exception` catches failures **per title**; a batch call converts this to **all-or-nothing** per batch. If title #17 crashes the batch helper, titles #1-16's updates and titles #18-N's updates all fail together. The threat surface is low (titles are sanitized, regex is escaped) but the observability changes. The brainstorm doesn't address this; the plan gate should mandate either (a) inner try/except per chunk inside the batch helper, or (b) document the behaviour change explicitly in CHANGELOG.

**Verdict:** MAJOR-failure-granularity-regression

### AC7 — `pages=` pre-loaded bundle threading
Precedent exists at `build_backlinks(wiki_dir=None, *, pages=None)` (`linker.py:114-169`). The contract signature is identical. Caller threads `load_all_pages` once — see `pipeline.py:1334` where `all_wiki_pages = load_all_pages(wiki_dir=effective_wiki_dir)` is already computed for `_find_affected_pages`. This can be threaded into `inject_wikilinks_batch(..., pages=all_wiki_pages)` with zero new disk I/O — confirmed no additional load.

One gap: `build_backlinks` expects dict entries with keys `id`, `content`. `inject_wikilinks_batch` will need `path` too (for atomic writes) — confirm the `load_all_pages` return shape actually includes `path`. From `pages.py` the dict has `id, path, content, ...` keys, so this composes correctly.

**Verdict:** PASS

### AC8 — two-phase write status field
The status enum `{pending, applied, failed}` is defensible (brainstorm D4), but the design skips a key question: what does `load_review_history` do with `status="pending"` rows that never flipped? Q4 in brainstorm.md defers this ("no TTL filter, keep visibility"), which is fine for auditor-facing reporting but breaks any downstream tool that assumes history entries are terminal. The cycle-19 plan should document that `load_review_history` consumers MUST handle `status="pending"` rows as "interrupted, unknown final state" rather than "applied".

The existing code at `refiner.py:230-238` writes a single history entry with `status="applied"` (line 236). Adding a pre-page pending write, then flipping, requires either (a) re-reading the history list + locating the row by `request_id`/timestamp + mutating in place, or (b) append-style transition tracking (rejected by brainstorm D4). Option (a) means the RMW window grows from load+append+save to load+locate+mutate+save — more CPU under the history lock (threat T4 amplifies).

The plan needs an unambiguous spec: status-flip uses correlation key `(timestamp, page_id)` OR a new `attempt_id` field. Without this, two concurrent refines on different pages could race to update each other's pending rows.

**Verdict:** MAJOR-flip-correlation-unspecified

### AC9 — pending flip inside history-lock
The logic is straightforward: acquire `file_lock(history_path)`, write pending, release; write page (under page_lock); re-acquire `file_lock(history_path)`, flip to applied/failed, release. But AC9 says **"the history file is RMW-locked across the page-write window"** which is different — it means *one* lock span covers both writes. This is the stronger liveness-harming semantic (T4 MEDIUM) that the threat model accepts.

Choose one: either re-acquire the lock (shorter critical section, higher audit-row staleness window) or hold through (T4 accepts). The brainstorm D8 implies holding through. Requirement text is ambiguous: "inside the same `file_lock(history_path)` window that wrote the pending entry". Plan-gate must lock the semantic.

**Verdict:** MAJOR-lock-span-ambiguity

### AC10 — lock order: history FIRST, page SECOND
**This is the blocker.** The current code at `review/refiner.py:85` has a docstring comment that reads literally: *"Lock-order: page_path acquired FIRST; history_path acquired SECOND (below)."* The H1 Phase-4.5 fix ordered page-FIRST precisely because the whole page lifecycle (read → validate frontmatter → compute diff → write) must serialize under `page_path`. History-second was justified as a cross-process audit trail that couldn't deadlock.

AC10 flips this to "history FIRST, page SECOND" with the justification that it preserves "existing two-lock chain" — but the existing chain is already page-first. The threat model T4 even admits "there is no existing caller holding history-lock and then requesting page-lock — the deadlock risk is null in the current call graph", meaning the flip is not motivated by a deadlock concern in the current codebase.

The **real** reason for the flip is AC8's pre-page pending write: if the page lock is acquired first, the pending-history write happens under *both* locks (history inside page), which is fine. The current page-first order already supports AC8 as long as the pending write is nested inside the page lock. There is **no technical requirement** to flip the order.

Flipping the order introduces a liveness regression (T4 MEDIUM, acknowledged in the threat doc) that doesn't need to exist. The brainstorm D8 offers no deadlock rationale; threat-model T4 attacker field says "N/A — self-inflicted liveness risk".

**Verdict:** BLOCKER-unjustified-lock-flip

### AC11 — `manifest_key_for` public alias
`_canonical_rel_path` (`compile/compiler.py:50`) is the existing computation — 8 lines, no dependencies. Exposing it via a public one-line alias (`manifest_key_for = _canonical_rel_path`) is mechanical. Q6 marks this keyword-only; Q14 keeps it out of `kb.__all__`. Both defensible.

**Verdict:** PASS

### AC12 — `manifest_key=` keyword-only
Additive kwarg with opaque-string semantics. T3 threat documents the risk: caller can inject any string as the dict key, but the key is never used as a filesystem path. The docstring-only mitigation is weak — if a plugin passes `"../../etc/passwd"`, manifest pollution is benign *today* because `.data/hashes.json` keys are never path-resolved. If a future caller ever `Path(manifest_key).resolve()` — RCE.

Plan-gate should add: AC12 MUST reject `manifest_key` values containing `..` or leading `/` at the `ingest_source` entry, even though today's code doesn't need it. Defense-in-depth, matching the cycle-14 path-traversal pattern.

**Verdict:** MAJOR-traversal-hardening-missing

### AC13 — `compile_wiki` threads canonical key
`compile_wiki` at `compiler.py:405` already computes `rel_path = _canonical_rel_path(source, raw_dir)` before entering the try block. Threading it into `ingest_source(source, wiki_dir=wiki_dir, manifest_key=rel_path, _skip_vector_rebuild=True)` is a one-line change. The existing exception path at `:433-436` also writes the manifest with `rel_path` — both paths must use the same key. Verified: both sites already use `rel_path` computed from the same `_canonical_rel_path` call at line 405.

Concern: the T-13 test specification "tilde-shortened Windows path" is awkward. Python's `Path.resolve()` already normalizes case and unfolds tildes on Windows. Constructing a test where `_canonical_rel_path` produces divergent keys would need `os.environ["USERPROFILE"]` manipulation, symlinks, or a mocked `resolve()`. The divergence claimed in the problem statement (§5: "Under Windows case differences or raw_dir overrides the same source produces two divergent manifest keys") is hard to reproduce in tests without filesystem-layer mocking. The plan should replace T-13's tilde-path setup with either (a) a direct unit test of `manifest_key_for(source, raw_dir) == manifest_key_for(source, raw_dir_alt_case)`, or (b) mark T-13 `skipif(sys.platform != "win32")` as brainstorm Q7 suggests.

**Verdict:** MAJOR-regression-test-hard-to-construct

### AC15 — MCP test monkeypatch migration
Enumeration confirms 13 sites / 7 files, no spooky patterns. Atomic commit (brainstorm D6, option c) is the correct approach. The `kb.mcp.core` refactor switches from `from kb.ingest.pipeline import ingest_source; ingest_source(...)` to `from kb.ingest import pipeline; pipeline.ingest_source(...)`, so test monkeypatches on `kb.ingest.pipeline.ingest_source` actually intercept the MCP call path. Cycle-18 L1 (snapshot-binding) is the governing principle.

One gotcha: `test_v0913_phase394.py:430` uses `monkeypatch.setattr(..., raising=False)` — after the migration, the attribute exists on `kb.ingest.pipeline`, so `raising=False` becomes unused. Drop it during migration to avoid a silent-failure mode (if a future rename kills the attribute, the test would silently pass).

Two cycle-18 L1 invocation patterns are in tension:
1. `from kb.ingest import pipeline; pipeline.ingest_source(...)` — mcp.core looks up `pipeline.ingest_source` at call time → tests patch `kb.ingest.pipeline.ingest_source` ✓
2. The existing `from kb.ingest.pipeline import ingest_source; ingest_source(...)` binds at import → tests must patch `kb.mcp.core.ingest_source`

The migration is from pattern 2 → pattern 1. This is correct and the plan should verify no intermediate import aliases are introduced.

**Verdict:** PASS

### AC16 — constant patches remain on `kb.mcp.core`

## Recommended amendments

- **AC10 (BLOCKER):** Withdraw the lock-order flip. The existing `page_path FIRST, history_path SECOND` order at `refiner.py:85` is correct for the current call graph (T4 confirms no deadlock risk). Nest the AC8 pending-write inside the existing page_lock; use `with file_lock(history_path)` as a sub-lock. Update AC10 test to assert **"existing order preserved"** with docstring rationale added.

- **AC2 (MAJOR):** Rewrite budget as `≤ U + 2M` where `U` = unmatched pages, `M` = matched pages. For T-2's 10-page scenario, explicitly construct pages so `M=0` (budget ≤ 10) or document the matched count and adjust.

- **AC6 (MAJOR):** Require inner try/except at batch-chunk granularity inside `inject_wikilinks_batch`. If chunk 3 of 5 raises, chunks 1/2/4/5 still complete. Document the failure-granularity contract in the docstring.

- **AC8 (MAJOR):** Specify the pending-row correlation key explicitly. Options: `(timestamp, page_id, content_length)` tuple OR a new `attempt_id = uuid4().hex[:8]` field. Plan-gate must lock one; the pending-flip logic hinges on it.

- **AC9 (MAJOR):** Disambiguate "inside the same file_lock(history_path) window": (a) release and re-acquire around page write, OR (b) hold across page write. Brainstorm D8 implies (b); requirement text reads ambiguously. Pick one.

- **AC12 (MAJOR):** Add input validation: reject `manifest_key` values containing `..`, leading `/`, `\x00`, or exceeding 512 chars at the `ingest_source` entry. Defense-in-depth; matches existing traversal-rejection pattern in `_validate_page_id`.

- **AC13 (MAJOR):** Replace T-13's tilde-path Windows test with a direct unit test: `manifest_key_for(source, raw_dir_A) == manifest_key_for(source, raw_dir_B)` where A and B differ only in case/resolution. Skip the windows-specific path-shortening scenario.

- **AC14 (MAJOR):** Before writing code, construct a concrete failing test case that demonstrates the prune divergence TODAY with non-default `raw_dir`. If no such case exists (cycle-17 AC1 may have already fixed it), remove AC14 from cycle 19 scope.

- **AC16 (MAJOR):** Add a behavioural test (not just docstring-extraction) that proves snapshot-binding: import `kb.mcp.core`, patch `kb.config.PROJECT_ROOT`, call an MCP tool, assert the pre-patch value was used. This is the real invariant.

- **AC17 (MAJOR):** Re-grep to confirm the file count (I count 6, not 12). Also correct the "autouse" claim in AC17/AC18 text — `tmp_kb_env` is a declarative fixture, not autouse.

- **AC1 (minor):** Clarify T-1 setup so pages have disjoint title mentions, avoiding accidental AC3-interference. No verdict change.

- **AC15 (minor):** Drop `raising=False` from `test_v0913_phase394.py:430` during migration so future attribute removal surfaces as a test failure.

- **AC19 (minor):** Add an explicit `assert not raised TimeoutError` in the e2e lock-count test to rule out stuck-lock flake.

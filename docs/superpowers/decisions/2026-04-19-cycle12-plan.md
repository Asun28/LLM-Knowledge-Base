## Commit ordering justification

1. `tmp_kb_env` lands first because custom-project and sanitizer regressions can safely use it only after the fixture exists and has its own leak test.
2. IO and config changes are independent leaf commits; landing them early gives their focused tests fast feedback without coupling to frontmatter or MCP work.
3. The frontmatter helper and the local `load_all_pages` migration are atomic in the same file, so the helper contract is proven before any cross-file migration depends on it.
4. The lint migration follows the helper and stays STRICT-4 to keep the cycle-12 blast radius aligned with the design gate.
5. The MCP entrypoint cluster crosses `src/kb/mcp/__init__.py`, `src/kb/mcp_server.py`, and `pyproject.toml`, but it is one logical commit because splitting it can create transient packaging or compatibility states.
6. The graph doc commit is independent and small; it can land before or after the MCP cluster without affecting runtime behavior.
7. The augment and sanitizer regression commits are tests-only but depend on the cycle fixture where they need write isolation.
8. `BACKLOG.md`, `CHANGELOG.md`, and `CLAUDE.md` ship last so stale backlog entries are deleted only after their corresponding source/test commits are green, and the changelog test count can reflect the final collection result.

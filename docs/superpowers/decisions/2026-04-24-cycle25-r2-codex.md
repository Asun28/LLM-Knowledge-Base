# Cycle 25 R2 Codex Design Review

## 1. Edge Cases Per Cluster

### Cluster A - `rebuild_indexes` `.tmp` awareness

- Claim: If `rebuild_indexes` tries to unlink `<vec_db>.tmp` while `rebuild_vector_index` is mid-build on Windows, the unlink can plausibly fail with `PermissionError`, so AC1 must preserve `result["vector"]["cleared"] = True` when only tmp cleanup fails. Evidence: `rebuild_vector_index` builds into `tmp_path` at `src/kb/query/embeddings.py:182` and `src/kb/query/embeddings.py:194`, replaces it at `src/kb/query/embeddings.py:201`, and AC1 condition says tmp unlink failure must not blank the main cleared status at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:58`.
- Claim: A stale `.tmp` left by a crashed vector rebuild is currently cleaned only by the next `rebuild_vector_index`, not by `rebuild_indexes`, so an operator can see confusing residue after a clean-slate command. Evidence: the problem statement says `rebuild_indexes` "only unlinks `<vec_db>`, not `<vec_db>.tmp`" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:10`, while current `rebuild_indexes` unlinks only `vector_path` at `src/kb/compile/compiler.py:623-628`.
- Claim: A caller-provided `vector_db` override makes the tmp sibling derivation ambiguous unless AC1 explicitly derives tmp from the effective `vector_path`, not always from `_vec_db_path(effective_wiki)`. Evidence: `rebuild_indexes` accepts `vector_db: Path | None = None` at `src/kb/compile/compiler.py:532` and assigns `vector_path = vector_db or _vec_db_path(effective_wiki)` at `src/kb/compile/compiler.py:599`.

### Cluster B - Vector dim-mismatch counter and guidance

- Claim: The module-level counter can undercount under true concurrent FastMCP worker threads if implemented as a bare `+= 1`, because the GIL does not make read-modify-write semantically atomic across bytecode scheduling. Evidence: AC4 requires a module-level `_dim_mismatches_seen` increment on every mismatch at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:38`, while current `VectorIndex.query` has no lock around the mismatch branch at `src/kb/query/embeddings.py:477-487`.
- Claim: The correctness impact of a missed increment is observability-only, not query-result corruption, because the mismatch path already returns `[]` before executing sqlite search. Evidence: current code returns `[]` immediately on dim mismatch at `src/kb/query/embeddings.py:487`, and the requirements describe the counter as "for observability/testing" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:38`.
- Claim: The warning remains once-per-instance even though the counter must increment per mismatch, so tests need to separate log spam suppression from event counting. Evidence: current `_dim_warned` is initialized per instance at `src/kb/query/embeddings.py:288` and gates `logger.warning` at `src/kb/query/embeddings.py:478-486`, while AC4 says the counter is "NOT once-per-instance" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:38`.

### Cluster C - `compile_wiki` in-progress marker

- Claim: Machine A running `kb compile` and machine B running `kb rebuild-indexes` concurrently can leave machine A's pre-marker erased, then later recreated as `failed:{pre_hash}` only if `ingest_source` raises. Evidence: `rebuild_indexes` unlinks the manifest under `file_lock(manifest_path, timeout=1.0)` at `src/kb/compile/compiler.py:611-614`, while the failure branch in `compile_wiki` reloads and writes `failed:{pre_hash}` under `file_lock(manifest_path)` at `src/kb/compile/compiler.py:465-469`.
- Claim: Machine B running the same `kb compile` two seconds after machine A can produce a false-positive stale-marker warning even when machine A is healthy, because AC7 scans once at entry and AC6 intentionally releases the manifest lock before `ingest_source`. Evidence: threat model T3 states a second compile can read the first process's `in_progress:{hash}` and warn at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:10`, and AC6 lock scope forbids holding the lock across `ingest_source` at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:62`.
- Claim: The marker cannot cover `content_hash(source)` failures under the current recommended timing, so a source that cannot be hashed remains a skipped error without `in_progress:` or `failed:` state. Evidence: current code computes `pre_hash = content_hash(source)` before entering the ingest try block and continues on `OSError` at `src/kb/compile/compiler.py:430-435`, and brainstorm Q5 recommends placing the marker after `pre_hash` is known at `docs/superpowers/decisions/2026-04-24-cycle25-brainstorm.md:51`.
- Claim: Full-mode tail pruning can delete unrelated `in_progress:` manifest entries whose raw files no longer exist, because the prune filter only protects `_template/` keys. Evidence: full-mode pruning deletes every non-template key whose `(prune_base / k).exists()` is false at `src/kb/compile/compiler.py:490-498`, and AC7 says stale markers should not be auto-deleted at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:46`.

### Cluster D - BACKLOG and CVE re-verification

- Claim: The CVE maintenance flow is process-sensitive because BACKLOG date edits are allowed only after both `pip-audit` and `pip index` checks confirm no fix. Evidence: AC9 requires `pip-audit --format=json`, both `pip index versions` commands, and "IF and ONLY IF" date edits at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:52`.
- Claim: The resolved BACKLOG deletion is coupled to a CHANGELOG entry, so implementation cannot treat AC10 as a standalone cleanup. Evidence: condition 8 says "BACKLOG deletion + CHANGELOG entry happen together in one commit" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:65`.

## 2. Failure-Mode Analysis Per Cluster

### Cluster A - partial state, recovery, observability

- Claim: If main vector DB unlink succeeds but tmp unlink fails, recovery is to rerun `kb rebuild-indexes` or `kb compile`, but observability depends on a compound error that preserves both paths. Evidence: AC1 requires compound `result["vector"]["error"]` for `vec_path` and `tmp_path` failures at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:30`, and `rebuild_indexes` already reports vector status through `result["vector"]` at `src/kb/compile/compiler.py:602-604`.
- Claim: If tmp unlink fails before or during an active rebuild, `rebuild_vector_index` itself will also attempt tmp cleanup on exception and re-raise, so AC1 should not swallow a persistent lock as success-only. Evidence: `rebuild_vector_index` unlinks tmp on build or replace exception at `src/kb/query/embeddings.py:202-210`, while its docstring says stale tmp is unlinked at entry and on exception at `src/kb/query/embeddings.py:125-131`.
- Claim: The audit log currently records only aggregate vector state, so an AC1 tmp-only failure needs to flow into `result["vector"]["error"]` if operators are to see it in `wiki/log.md`. Evidence: `rebuild_indexes` formats the audit message from `result["vector"]["cleared"]` or `result["vector"]["error"]` at `src/kb/compile/compiler.py:657-663`.

### Cluster B - partial state, recovery, observability

- Claim: Dim mismatch recovery is manual by design, so the warning text is the only remediation path visible at query time. Evidence: the non-goal rejects auto-rebuild and says the operator runs `kb rebuild-indexes` at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:20`, while current query just logs and returns empty at `src/kb/query/embeddings.py:479-487`.
- Claim: The counter is process-local and disappears on restart, so it supports live diagnosis and tests but not historical incident review. Evidence: the threat model says the counter has "no network/disk persistence" at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:8`.
- Claim: A counter increment placed inside the `if not self._dim_warned` block would violate AC4 by counting only the first mismatch per `VectorIndex` instance. Evidence: current warning is gated by `_dim_warned` at `src/kb/query/embeddings.py:478-486`, and AC4 requires incrementing on every query that detects a mismatch at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:38`.

### Cluster C - partial state, recovery, observability

- Claim: On normal Python exceptions after marker write, the final manifest should show `failed:{pre_hash}`, not `in_progress:{pre_hash}`, so recovery is retry via incremental compile. Evidence: AC6 says the existing exception block replaces the marker with `failed:{pre_hash}` at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:44`, and current exception code writes `failed:{pre_hash}` at `src/kb/compile/compiler.py:465-469`.
- Claim: On hard kill after marker write but before `ingest_source` overwrites it, recovery is manual investigation because AC7 explicitly warns and does not delete. Evidence: AC6 says `in_progress:` only survives hard kill or abort between pre-marker and `ingest_source` at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:44`, and AC7 says "Do NOT auto-delete" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:46`.
- Claim: Observability is split between Python logs for stale markers and `wiki/log.md` for compile summaries, so stale marker events may be absent from the wiki ledger. Evidence: threat model says AC7 stale-marker detection emits `logger.warning` only and "does NOT write log.md" at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:31`, while `compile_wiki` appends only the final compile summary at `src/kb/compile/compiler.py:514-523`.

### Cluster D - partial state, recovery, observability

- Claim: If `pip-audit` finds a fix version, the correct recovery is a dependency bump rather than date-stamp maintenance. Evidence: AC9 says "If either shows a fix, run Step-11.5 `fix(deps)` commit bumping `requirements.txt`" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:52`.
- Claim: If BACKLOG deletion ships without CHANGELOG updates, the repository violates its own lifecycle convention even though runtime behavior is unaffected. Evidence: AC10 removes the BACKLOG entry at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:54`, and condition 8 requires the CHANGELOG entry in the same commit at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:65`.

## 3. Integration

- Claim: Adding `.tmp` unlink to `rebuild_indexes` changes only cleanup breadth, not the public return schema, if errors remain inside `result["vector"]["error"]`. Evidence: the current return schema contains `manifest`, `vector`, `caches_cleared`, and `audit_written` at `src/kb/compile/compiler.py:566-570`, and AC1 specifies adding tmp errors to `result["vector"]["error"]` at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:30`.
- Claim: No caller should rely on `.tmp` surviving `rebuild_indexes`, because the requirements define surviving tmp files as operator-confusing hygiene residue rather than supported state. Evidence: the problem statement says the `.tmp` survives until next rebuild and is "not a correctness bug ... but a hygiene gap" at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:10`.
- Claim: The counter does not need to be part of `mcp/app.py` instructions for cycle 25 unless the Step 5 gate wants external observability, because AC4 exposes only an in-process getter and CLAUDE says there is no MCP rebuild-indexes tool today. ## 4. Security Deltas

- Claim: T4 counter overflow is correctly dismissed for Python integer overflow, but not for unbounded process-local growth in a pathological long-lived process. Evidence: the threat model says Python 3 ints are arbitrary-precision and only eventual memory consumption remains at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:10`.
- Claim: The T5 path leak is real because the required warning includes both vector DB path and wiki-dir path, and this conflicts with the broader StorageError redaction principle if logs are aggregated. Evidence: AC3's exact format includes `%s` for the vector DB path and `%s` for `--wiki-dir` at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:36`, while CLAUDE says `StorageError` redacts raw filesystem paths so log aggregators never see them at `CLAUDE.md:190`.
- Claim: The T5 risk is bounded by existing scope because the change adds no new sink beyond an existing `logger.warning`. Evidence: threat model analysis says AC3 "changes only the content of an existing `logger.warning` and does NOT introduce any new sink" at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:8`.
- Claim: The `.tmp` unlink does not materially add path traversal risk when derived from the validated vector path, but a custom `vector_db` override should receive the same scrutiny as the existing override. Evidence: `rebuild_indexes` validates `wiki_dir` under `PROJECT_ROOT` at `src/kb/compile/compiler.py:585-594`, but `vector_db` is accepted directly into `vector_path` at `src/kb/compile/compiler.py:599`.
- Claim: `in_progress:` markers do not add secret material to the manifest because they contain a content hash and reuse the plain-text source path key. Evidence: the threat model classifies `f"in_progress:{pre_hash}"` as public-derivable and says `rel_path` is already stored in plain text at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:26`.

## 5. Perf

- Claim: The counter increment overhead is a single module global read-modify-write on the mismatch path, so normal matched vector queries do not pay if the increment is inside `len(query_vec) != stored_dim`. Evidence: current code branches on dim mismatch at `src/kb/query/embeddings.py:477` and otherwise proceeds to the sqlite query at `src/kb/query/embeddings.py:489-499`.
- Claim: Under heavy mismatch load, an unlocked global increment is cheaper than a lock but provides approximate counts under multi-threading. Evidence: brainstorm B1 recommends a module-level counter and getter at `docs/superpowers/decisions/2026-04-24-cycle25-brainstorm.md:20-21`, while no lock is specified in AC4 at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:38`.
- Claim: Adding a lock for exact counts would likely cost more than the sqlite early-return path it observes, so exactness should be required only if Step 5 treats the counter as billing-grade telemetry. Evidence: current mismatch path returns before the `conn.execute(...)` query at `src/kb/query/embeddings.py:477-489`, and threat model classifies the counter as in-process observability rather than persisted telemetry at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:8`.
- Claim: AC7's stale marker scan is cheap for ordinary manifests but has unbounded linear cost in manifest size because it scans stored manifest entries once per `compile_wiki` invocation. Evidence: AC7 requires the check run once at function top at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:46`, and `load_manifest` returns the whole JSON object from disk at `src/kb/compile/compiler.py:82-88`.

Evidence: AC4 requires every mismatch query be counted at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:38`, and current `VectorIndex.query` has no synchronization around the mismatch branch at `src/kb/query/embeddings.py:477-487`.

3. Claim: Step 5 must choose whether live `in_progress:` warnings are acceptable noise or require a grace-window check before warning. Evidence: AC7 requires a one-time entry scan at `docs/superpowers/decisions/2026-04-24-cycle25-requirements.md:46`, and threat model T3 acknowledges false positives from a legitimate second compile at `docs/superpowers/decisions/2026-04-24-cycle25-threat-model.md:44`.

## Executive Summary

Cycle 25 is narrowly scoped and generally safe, but three details need gate-level decisions before implementation. The `.tmp` cleanup should be implemented as a separate best-effort unlink after the main vector DB unlink so a locked tmp file on Windows does not erase a successful `vector.cleared` status. The spec should also say tmp is derived from the effective `vector_path`, because `rebuild_indexes` accepts a `vector_db` override and callers using that override will expect its sibling tmp to be cleaned too.

The dim-mismatch counter is useful, but the current wording "every query" is stronger than an unlocked module-level `int += 1` under FastMCP worker threads. Python prevents memory corruption, not necessarily lost read-modify-write updates. If the counter is diagnostic, approximate counts are acceptable and cheap; if it is intended as exact telemetry, add a lock and test concurrent mismatches. The warning text intentionally logs absolute paths, which is a bounded but real path-disclosure delta compared with the StorageError redaction convention.

The `in_progress:` marker design is the right small fix for partial ingest state, but it will warn on legitimate concurrent compiles. That is non-destructive, yet the word "stale" will be noisy unless the gate accepts it explicitly or adds a grace window. Full-mode pruning also deserves attention because it can delete non-template manifest keys, including stale markers for missing raw files, despite AC7 saying operators decide marker cleanup. CVE and BACKLOG criteria are procedural and clear. These decisions are small, but they define whether cycle 25 is diagnostic hygiene or operator-facing observability.

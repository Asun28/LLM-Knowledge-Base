# Cycle 24 — R2 Codex Design Review

**Date:** 2026-04-24  
**Reviewer:** R2 / Codex  
**Scope:** Edge-case, failure-mode, integration, security, and performance analysis for AC1-AC15.

## 1. Edge Cases Per Cluster

### Cluster A — Evidence-Trail Reliability

- `_write_wiki_page(exclusive=True)` is not an `atomic_text_write` call today. AC1 says "single `atomic_text_write` call" for the new-page path, but the production ingest path uses `exclusive=True` for summaries/items. If the literal AC is implemented only in the non-exclusive branch, the actual ingest-created pages keep the two-write race.
- `_write_wiki_page` retry/collision behavior can double-create evidence entries if the implementation keeps any fallback `append_evidence_trail` after inline rendering. A summary-collision retry pivots to `_update_existing_page`; that update path must not add an "Initial extraction" entry after a page already has the inline first entry.
- Inline rendering must reuse `format_evidence_entry` or an equivalent pipe-neutralizing path. Re-implementing the row with a raw f-string regresses the existing pipe escape for `source_ref` or action.
- Crash during `exclusive=True` raw `os.write` after evidence is appended to the rendered payload still leaves the current zero-byte poison cleanup path. The cleanup must continue to unlink the reserved file on any partial write. A larger payload raises the chance of short writes or ENOSPC during the single write.
- `_update_existing_page` AC2 wraps only `OSError`. `append_evidence_trail` can also fail with `UnicodeDecodeError`, malformed lock content `OSError`, or programmer errors in formatting. The requirement is clear for storage failures, but failure taxonomy will remain inconsistent for non-`OSError` evidence append failures.
- If `_update_existing_page_body` returns `False` because the frontmatter source is already present but the page lacks an evidence-trail entry due to an earlier crash, AC2 does not repair the missing trail. The system remains provenance-incomplete forever unless lint or a manual repair notices it.
- Frontmatter-planted sentinel is not explicitly covered. AC15 mentions a body sentinel before the real trail. Current `append_evidence_trail` searches the full file, and YAML frontmatter is before the body and before `## Evidence Trail`; an attacker-controlled title/source-like field containing `<!-- evidence-trail:begin -->` would divert insertion even earlier.
- Existing pages with `## Evidence Trail` inside fenced code before the real trail can still confuse a naive header-first regex unless the implementation masks fenced blocks like `_update_existing_page_body` does for `## References`.

### Cluster B — Vector-Index Atomic Rebuild

- `os.replace(tmp, vec)` is atomic only for the directory entry replacement. It does not guarantee durability unless the tmp DB has committed, all sqlite handles are closed, and the parent directory metadata is flushed. The ACs do not require parent-dir fsync, so power-loss after replace can still lose the new directory entry on some filesystems.
- On Windows, cached `VectorIndex._conn` handles can block `os.replace`. The requirements condition says `_index_cache.pop` before replace, but pop does not close `VectorIndex._conn`; it only removes the cached reference. Any thread holding the object, or the popped object before GC closes the connection, can still pin the file.
- Cross-process rebuild remains unsafe. `_rebuild_lock` is a `threading.Lock`, not a file lock. Two processes can both unlink the same `.tmp`, build into the same tmp path, and race `os.replace`. The later replacer wins; if either build fails, it can delete or corrupt the other's tmp.
- AC6 stale tmp unlink at entry is underspecified relative to gates. If it happens after `_hybrid_available`, after mtime skip, or inside `_rebuild_lock`, stale tmp files persist when rebuild is skipped. The brainstorm calls for first statement behavior; the AC text only says "at entry".
- Empty-page rebuild is a separate path. Today `VectorIndex.build([])` creates an empty DB directly at production path. AC5 must cover the no-pages branch, not only the embeddings branch.
- `VectorIndex.build(entries, db_path=tmp)` must reset the correct instance metadata. If `self._stored_dim` is set while building a different `db_path`, the object associated with `vec_path` may claim a dim for a DB it did not build.
- Crash after tmp build but before replace leaves a valid tmp and old production DB. That is safe for query, but operationally ambiguous: the next run unlinks the tmp without knowing whether it was a complete newer build or a partial build.
- `rebuild_indexes` from cycle 23 unlinks the vector DB directly and does not clear `_index_cache`; after AC5, it still will not remove `.tmp` unless explicitly updated. Operator recovery can therefore leave stale cached readers and stale tmp artifacts.

### Cluster C — `file_lock` Exponential Backoff

- Exponential backoff is unfair by construction. New waiters start at 10ms while old waiters may be sleeping at the cap; late arrivals can repeatedly beat long-waiting callers.
- The stale-lock steal branches currently sleep `LOCK_POLL_INTERVAL` after unlinking a dead-PID lock. If AC9 changes only the normal `FileExistsError` path, stale-lock recovery retains the 50ms floor and the test may miss it.
- A very low monkeypatched `LOCK_POLL_INTERVAL` cap, such as `0.001`, can create busy-poll behavior under adversarial contention. Compatibility is useful for tests, but production code can also monkeypatch the module and create a CPU DoS.
- `LOCK_TIMEOUT_SECONDS` is still 5 seconds. Under the proposed 100ms cap, the number of attempts is lower than fixed 50ms after the first few retries. That changes stale-lock detection latency and can increase timeout odds under high contention.
- Sleep-spy tests are easy to make vacuous. A seeded lock file can transition to stale-lock logic instead of ordinary contention; a multiprocessing test can pass by never actually sleeping. AC10 needs a deterministic contention harness with several consecutive `FileExistsError` iterations.
- `PermissionError` must remain fail-fast. A broad refactor that wraps all `OSError` in retry/backoff would re-open the cycle-3 corruption issue described in `io.py`.

### Cluster D — BACKLOG Maintenance

- CVE re-checks depend on live package-index and advisory state. A date bump without recording command output and `fix_versions` state can become stale the same day.
- AC12 says update diskcache re-check date to 2026-04-23, but the current execution date is 2026-04-24 in this environment. A same-day-or-later cycle gate should not deliberately write an already stale date.
- If `pip index versions diskcache` or `pip-audit` cannot run because network is restricted or tooling is absent, the AC text does not say whether the doc edit must fail closed, preserve the old date, or record "unable to verify".
- Deleting the multiprocessing backlog entry can hide residual file-lock gaps. Cycle 23 shipped cross-process regression coverage, but AC9 changes timing semantics; the deleted backlog item may have to be replaced by a narrower fairness/contention item rather than simply removed.
- CHANGELOG and BACKLOG update requirements do not mention `CHANGELOG-history.md`, but the blast-radius section does. A partial doc update can violate the repo's doc lifecycle convention in `CLAUDE.md`.

### Cluster E — Sentinel-Anchor Hardening

- Anchoring to the first `## Evidence Trail` header is necessary but insufficient if the first header appears in frontmatter-adjacent text or a fenced code block. Existing pipeline code already masks code blocks for references; evidence should consider the same class of bug.
- Searching "within the bytes following that header" can still find a planted sentinel in content after the real header but before the machine sentinel. The search should be limited to the actual Evidence Trail section span, not the entire tail of the document.
- Header variants are undefined. Lint tolerates trailing horizontal whitespace via `^## Evidence Trail[ \t]*\r?\n`; AC14 says "FIRST `## Evidence Trail` section header" but does not state whether `## Evidence Trail  `, CRLF, or `### Evidence Trail` are valid.
- If a page has a header but no sentinel, inserting the sentinel at `header.end()` can place new entries before legitimate old entries. That is probably intended for migration, but the requirement should state the ordering for header-only legacy pages.
- If a page has no header but has a sentinel, AC14's fallback behavior is not specified. Current behavior uses the sentinel wherever it appears. A hardening patch must decide whether to ignore sentinel-without-header and create a fresh section, or preserve legacy sentinel-only pages.
- AC1 makes every new page emit the sentinel at first write. That increases the chance that copied page templates, examples, or user-authored content include the exact sentinel string. The hardening must treat the sentinel as structural only inside the real section.

## 2. Failure-Mode / Recovery Analysis

### Cluster A

- Partial state: new page body exists without evidence trail if AC1 misses the exclusive branch or if write failure cleanup is incomplete. Recovery is manual unless a future repair/lint command inserts missing initial evidence. Observable only if a test or lint checks for missing `## Evidence Trail`; normal ingest success can hide it.
- Partial state: update body/frontmatter source is written but evidence append fails. AC2 makes this a `StorageError(kind="evidence_trail_append_failure")`; recovery is manual re-ingest or manual evidence repair. Observable through the exception at direct call sites and through ingest/compile warning or failure telemetry if callers do not swallow it.
- Partial state: body-planted/frontmatter-planted sentinel receives future entries outside the trail. Recovery is manual content surgery. Observable only by careful file inspection or a dedicated lint check; current trail readers will silently miss the misplaced entry.

### Cluster B

- Partial state: crash during tmp build leaves old production DB plus `.tmp`. Recovery is automatic only if AC6 unlinks tmp before every future rebuild path, including skipped rebuilds if that is the chosen contract. Observable only if logging is added; AC6 does not require telemetry.
- Partial state: `os.replace` fails on Windows due to open cached connection. Production DB remains old, tmp may remain complete. Recovery is manual close/restart or rerun after cache/connection release. In ingest/compile, this is likely only a warning because rebuild failures are swallowed.
- Partial state: crash after replace but before cache eviction/log. Production DB is new, cached readers may serve old results until process restart or cache miss. Recovery is automatic only for new `get_vector_index` calls after cache pop; not automatic for already-held objects. Observable through stale search results, not structured telemetry.
- Partial state: empty wiki rebuild creates or replaces an empty DB. Recovery is automatic on next rebuild with pages. Observable through "0 entries" log if that log is after successful replace.

### Cluster C

- Partial state: no persistent data change; failures are timing failures. A waiter can time out under contention even though the lock is healthy. Recovery is automatic retry by caller if caller has retry logic; most callers surface or log the exception.
- Partial state: corrupted lock content still fails closed. Recovery is manual deletion after investigation. Observable as `OSError` with lock path in message; note this can leak local path information today.
- Partial state: stale lock from dead process is removed and acquisition retries. Recovery is automatic. Observable only indirectly through wait time; no log line.

### Cluster D

- Partial state: docs claim CVEs were rechecked but commands did not actually run or failed. Recovery is manual re-audit. Observable only in review if the command transcript is included in changelog/history or PR notes.
- Partial state: BACKLOG entry deleted while residual risk remains. Recovery is manual re-add. Observable only by later design review or missing deferred-to-backlog grep.

### Cluster E

- Partial state: new entries inserted under wrong sentinel. Recovery is manual relocation and possibly deletion of forged sentinel text. Observable only by a targeted test/lint; regular queries and stats are unlikely to detect it.
- Partial state: hardening ignores legacy sentinel-only pages and creates a second Evidence Trail section. Recovery is manual merge. Observable by duplicate header grep or lint if added.
- Partial state: regex hardening fails on CRLF/trailing-space headers and appends a fresh section at EOF. Recovery is manual merge. Observable by duplicate trail sections.

## 3. Integration Analysis

- AC1 changes every newly ingested wiki page's first-write bytes. Downstream `load_all_pages`, `kb_query`, and BM25 indexing will now see evidence text immediately rather than after the append call; final content shape should be the same, but tests spying on `atomic_text_write` byte identity from cycle 20 will break and must be intentionally updated.
- AC1 must preserve frontmatter ordering from `frontmatter.dumps(..., sort_keys=False)`. Template consumers and YAML-diff tools rely on stable key order per `CLAUDE.md`.
- AC1 section placement interacts with `kb.query.dedup._TRAIL_SECTION_RE`, which strips everything from `## Evidence Trail` or `## References` onward. If a malformed or forged Evidence Trail heading appears before substantive content, dedup tokenization can discard real content.
- AC2 introduces a typed `StorageError` from `_update_existing_page`. `compile_wiki` and `ingest_source` broad exception boundaries should continue to catch it, but telemetry must preserve `kind` through `str(exc)` and avoid `repr(exc)` or `exc.path`.
- AC5 touches `kb_query` through `_index_cache` and persistent sqlite connections. A rebuilt DB does not guarantee already-cached `VectorIndex` instances stop serving old data.
- AC5 interacts directly with cycle-23 `rebuild_indexes`: that helper deletes the production DB but not `.tmp` and not `_index_cache`. If cycle 24 treats `.tmp` as part of vector index lifecycle, `rebuild_indexes` should either delete it or document that the next rebuild cleans it.
- AC5 also touches compile and ingest tail behavior. Both call `rebuild_vector_index` and currently log-and-continue on failure, so a broken atomic rebuild will not fail ingest/compile but will degrade query.
- AC7 is an internal API change, but tests and any local plugin importing `VectorIndex.build(entries)` can break if the kwarg is positional-only or if the signature change is not backward compatible.
- AC9 affects every subsystem using `file_lock`: evidence trails, manifests, feedback/verdict stores, refiner history, augment rate state, wiki log writes, and rebuild-index manifest deletion. It is a shared concurrency primitive, not a local optimization.
- AC11-AC13 are docs-only but integrate with the repo lifecycle rule: resolved backlog items are deleted, and changelog/history must remain newest-first and compact/full respectively.
- AC14 should align with existing lint section anchoring. `check_authored_by_drift` already scopes to the Evidence Trail span; a different regex tolerance in `append_evidence_trail` creates disagreement between writer and linter.

## 4. Security Deltas

- AC1 does not create a new executable injection vector by itself, but it increases the importance of evidence row escaping. `source_ref` and action must go through pipe neutralization; otherwise a malicious filename can forge apparent columns or entries in rendered markdown.
- AC1 can expose a structural injection vector through rendered content if an attacker can cause `## Evidence Trail` or the sentinel to appear in the page body before the machine section. AC14 addresses the sentinel case, but header-in-code/frontmatter cases remain underspecified.
- AC2 is safe only if `StorageError(kind=..., path=...)` is stringified. Logging `err.path`, using `repr(err)`, or wrapping it into a generic message containing the path reintroduces absolute path leakage.
- AC5 can leak path information through raw `OSError` messages from `os.replace`, stale tmp unlink, or sqlite open failures. Since ingest/compile log `"%s"` on the exception, any non-`StorageError` filesystem exception may include `.data/vector_index.db` absolute paths.
- AC5 creates a new `.tmp` DB artifact containing page IDs and embeddings. The data is not more sensitive than the production DB, but it increases the window where a scanner/backup/sync tool can capture partial derived state.
- AC5/AC6 stale tmp unlink is a deletion operation. If `vec_path` can ever be attacker-directed through a symlinked or malicious `wiki_dir.parent`, the tmp unlink target needs the same containment assumptions as `_vec_db_path`.
- AC9 can create DoS under adversarial contention. Many processes can force waiters into repeated sleeps and eventual timeouts; very low monkeypatched caps can also create high CPU wakeups. There is no fair queue or cross-process admission control.
- AC9 error messages include lock paths. Existing behavior already does this; changing retry logic is a chance to avoid widening path disclosure in new exceptions.
- AC12/AC13 require network/package-audit commands. Running these in an unrestricted environment can disclose package inventory to package indexes/audit services; this is existing operational behavior but should be acknowledged in audit notes.
- AC14 reduces sentinel injection risk, but regex-based anchoring can itself be bypassed if markdown constructs are not parsed or masked. Treat this as format hardening, not a full parser-level security boundary.

## 5. Performance Analysis

### Cluster A

- AC1 should be faster for new pages: one disk write and one lock acquisition instead of body write plus `append_evidence_trail` read/lock/write. Exclusive writes get a larger single payload but avoid a second RMW cycle.
- AC2 is performance-neutral on success. On failure it adds exception wrapping only.
- Risk: if AC1 chooses a helper that renders then reparses page content to insert evidence, the performance gain disappears. It should concatenate a canonical evidence section before the write.

### Cluster B

- AC5 is neutral to slightly slower: full sqlite build still happens, but now into `.tmp`, followed by `os.replace`. The replace is cheap; the real cost is unchanged embedding/model work.
- AC6 adds one `unlink(missing_ok=True)` per rebuild invocation. Negligible unless called frequently on network/cloud-synced `.data`.
- AC7 has no meaningful runtime cost.
- AC8 crash-simulation tests may be slow if they load real model2vec. They should stub embeddings/model and exercise rebuild mechanics, not model inference.

### Cluster C

- AC9 improves low-contention latency: first retry can be 10ms instead of 50ms.
- Under sustained contention with a 50ms cap, the schedule is same-or-faster than the old fixed 50ms after the first retries. With a 100ms cap, worst-case polling frequency is lower but individual wait latency can increase.
- High waiter counts may see lower syscall pressure but higher tail latency and weaker fairness.

### Cluster D

- Runtime impact is none. Developer-cycle impact depends on `pip-audit` and `pip index` network latency.

### Cluster E

- AC14 adds one or a few regex scans over page content per evidence append. This is negligible compared with disk I/O and LLM ingest.
- If implemented with fenced-code masking, cost is still linear in page size and acceptable. It is safer than repeated unbounded `str.index` scans over ambiguous regions.

## 7. Top 3 Open Questions For Step 5 Decision Gate

1. Should AC1 preserve the `exclusive=True` raw `os.open(O_EXCL)` reservation path with inline evidence, or should that branch be rewritten around `atomic_text_write` despite losing the exact reservation semantics?
2. What is the required Windows-safe cache behavior for AC5: is `_index_cache.pop` enough, or must the implementation explicitly close cached `VectorIndex._conn` handles before `os.replace`?
3. What exact compatibility contract should AC9 pin for `LOCK_POLL_INTERVAL`: default cap at 50ms, new cap at 100ms, monkeypatch cap for all sleeps, or legacy fallback mode?

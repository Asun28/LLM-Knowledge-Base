## 5. New Questions / Gaps Brainstorm Missed

### Q7 — AC3 message path ambiguity

**AC3 format literal:**
```
"Vector index dim mismatch: query=%d vs stored=%d at %s. Run 'kb rebuild-indexes --wiki-dir %s' to realign, OR ignore if BM25-only search is intended."
```

The two `%s` placeholders are both documented as "path to the vector DB" in §AC3 text but the second is the argument to `--wiki-dir`. `VectorIndex.__init__` only stores `db_path` (embeddings.py:283). `_vec_db_path(wiki_dir) = wiki_dir.parent / ".data" / "vector_index.db"`, so `wiki_dir = db_path.parent.parent`. The second `%s` should pass `self.db_path.parent.parent` — OR the message format should be reframed so only `db_path` appears (e.g., `"Run 'kb rebuild-indexes' from %s to realign" % self.db_path.parent.parent`, or drop the `--wiki-dir` flag entirely and let operators rely on the default). Requirements as written would produce an incorrect operator command like `kb rebuild-indexes --wiki-dir D:\...\.data\vector_index.db`, which `kb rebuild-indexes` would reject via `ValidationError("wiki_dir must be inside project root")` at compiler.py:588 (`.data/vector_index.db` is INSIDE project root but is NOT a wiki dir — there's no containment check against "is a wiki dir" so the unlink would proceed against a mis-targeted path). **Recommend:** require the design to specify `db_path.parent.parent` explicitly, OR drop the `--wiki-dir` substitution and keep just the default-scoped `kb rebuild-indexes`. This is a single-word fix in requirements text, not a scope change.

### Q8 — `in_progress:` marker and `find_changed_sources` re-entry

`find_changed_sources` at compiler.py:179 filters with `stored.startswith("failed:") or stored != current_hash`. An `in_progress:{pre_hash}` left by a hard-killed prior run will NOT start with `"failed:"` AND will not equal the current hash (different value space), so the source will be flagged as `changed_sources` and re-ingested. This is the correct recovery behaviour — good. But the AC7 stale-marker LOG will ALSO fire at loop entry for the same source, so the operator sees both a warning AND a re-ingest. That's intentional per AC7 ("operator decides") but worth noting in CLAUDE.md's Evidence Trail / compile_wiki notes: **"A stale `in_progress:` marker does not block re-ingestion — the source will re-process normally on the next `kb compile` invocation, with a warning naming the source."** Not a design change; a CLAUDE.md note for doc-sync. **DEFER to Step 12 doc-update.**

### Q9 — AC6 pre-marker lock + AC7 scan lock — are both `file_lock`-protected?

AC7 reads the manifest at loop entry to find `in_progress:` prefixes. The requirement text does NOT specify whether this read holds `file_lock`. If another process writes the manifest between our `load_manifest` and our iteration, we may miss a just-written `in_progress:` (benign — the other process will log its own warning on its own loop entry) OR we may see a `in_progress:` the other process just cleared (false-positive warning). Since cycle-17 AC3 established that RMW operations hold `file_lock` but read-only scans may not, AC7's scan is legitimately lock-free. **AGREE with this design, but recommend the implementation explicitly comment "intentional lock-free read — benign race per threat-model T3."** Doc-only clarification.

### Q10 — Counter reset between tests running in the same process

`_dim_mismatches_seen` is module-level and persists across test functions within a session. If two tests both construct a VectorIndex with mismatched dim and call `query`, the counter accumulates across both. CONDITION 3 already says "Reset helper is NOT required; tests rely on monotonicity." The monotonic-delta pattern `pre = get(); ...; post = get(); assert post - pre == 1` is standard pytest hygiene; the author should specify this idiom in the AC5 test spec to prevent a reviewer from demanding a reset helper. **Minor — test-spec clarification only.**

### Q11 — Threat-model T5 (`logger.warning` leaks absolute path) — mitigation language

Threat model accepts T5 (absolute-path leak in `logger.warning` for the dim-mismatch case) under cycle-20 L3 "developer-log is an accepted boundary for non-persistent sinks". AC3's new format string adds a SECOND absolute path reference (the `--wiki-dir %s`). T5's acceptance should be explicitly re-affirmed for the amplified surface, ideally in the threat model's T5 finding rather than implicitly. **Recommend:** threat-model.md's T5 paragraph could add a sentence noting that AC3 doubles the path-leak surface; current acceptance still holds (still developer-log), but the author should acknowledge the 2× multiplier for audit trail. **Minor — threat-model text addition, not a design change.**

## Analysis

The cycle-25 scope is unusually tight: 10 ACs, 4 of which (AC2, AC5, AC8, AC9) are tests + maintenance, and the remaining 6 (AC1, AC3, AC4, AC6, AC7, AC10) make additive changes to code sites that already exist with compatible contracts. Every cited line number in the requirements is either exact (e.g., `compile_wiki` at 425-471 — actual block is 425-471) or off by 2-3 lines from drift since the design was written (AC1 cites 620-628 but the vector-DB block is 623-628). None of the line references point at code that has been deleted or moved. The symbol verification table reveals zero ghost citations. This is the cleanest design handoff I have reviewed in the cycle-20 through cycle-25 arc.

Two concerns warrant author attention before Step-6 planning. First, AC3's warning-message format has a latent path bug: the second `%s` is the argument to `--wiki-dir`, but `VectorIndex` only stores `db_path`, so naively passing `self.db_path` produces an operator command that points at the `.db` file rather than the wiki directory. The fix is a one-word change (pass `self.db_path.parent.parent`), but the design text should specify this to avoid a round-trip in Step 9 implementation. Second, AC6's `in_progress:` marker shares the `manifest[rel_path]` slot with `failed:{hash}` and real content hashes, and the existing `find_changed_sources` filter at line 179 correctly treats it as "changed" (re-ingest eligible) — I verified the path semantics and the design is self-consistent, but CLAUDE.md's `compile_wiki` note should explicitly call out the three-state-value-space (`<hash>` / `failed:<hash>` / `in_progress:<hash>`) so a future cycle does not add a fourth state without revisiting the filter. That is a Step-12 documentation task, not a design blocker.
